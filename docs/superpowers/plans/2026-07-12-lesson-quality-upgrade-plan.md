# 개발계획서 — 차시 품질 대개편 (목표·활동·모의대화)

> 설계서: `docs/superpowers/specs/2026-07-12-lesson-quality-upgrade-design.md`
> 이 문서는 "무엇을, 어떤 순서로, 누가, 언제 끝났다고 보는지"를 한눈에 보기 위한 계획서입니다.

---

## 전체 그림

| 단계 | 하는 일 | 작업 방식 | 결과물 |
|---|---|---|---|
| **P1** | 학습목표 68차시 전부 쉬운 행동문으로 수정 | Sonnet 작업자 × 지침서 (`goal-rewrite-guide.md`) | 단원별 커밋 6개 |
| **P2** | 스마트폰 프레임 개발 + 모의대화 50회 적용 | 1회성 개발 (서브에이전트 + 리뷰) | PhoneFrame 컴포넌트, 커밋 1~2개 |
| **P3** | 미션 엔진 개발 (컨테이너 + 블록 9종 + 인쇄) | 1회성 개발 (서브에이전트 + 리뷰) | mission/ 컴포넌트 일습, 커밋 3~4개 |
| **P4** | **단원 1 파일럿** — 미션 11개 콘텐츠 작성 + 차시 재구성 | Sonnet 작업자 × 지침서 (`mission-authoring-guide.md`) | 커밋 2~3개 |
| **P5** | 사용자 확인 게이트 — 단원 1을 직접 써보고 승인 | 사용자 | 승인 or 수정 지시 |
| **P6** | 단원 2~6 확장 (미션 57개) | Sonnet 작업자 × 지침서 (단원별 반복) | 단원별 커밋 5개 |

순서는 P1 → P2 → P3 → P4 → P5 → P6. P1·P2는 서로 독립이라 병행 가능.

---

## P1 — 학습목표 수정 (콘텐츠만, 코드 없음)

- **대상**: `src/data/lessons/hard/m1.ts ~ m6.ts`의 goal 3종 × 68차시 = 204문장.
- **작업자에게 주는 것**: `docs/worker-guides/goal-rewrite-guide.md` 하나면 됨 (규칙 + 좋은 예/나쁜 예 + 절차 + 체크리스트 포함).
- **커밋 단위**: 단원당 1커밋 (총 6개). 커밋마다 lint·encoding 통과 필수.
- **검증**: 단원마다 리뷰어(별도 서브에이전트)가 체크리스트로 전 차시 검사 → 지적 나오면 수정 후 재검.
- **끝났다는 기준**: 68차시 전부 ① 행동문 ② 그 차시 활동과 일치 ③ 어려운 낱말 없음(핵심 용어 1개 예외) ④ 세 난이도 길이 차이 유지.

## P2 — 폰 프레임 (1회성 개발)

- **만들 것**: `src/components/PhoneFrame.tsx` (폰 외형 + 채팅 UI + 타이핑 연출).
- **고칠 곳**: `LessonView.tsx`의 renderSimAI (38회 전부 자동 적용), `RealAIStep.tsx` (12회 자동 적용).
- **데이터 수정 없음** — 기존 prompt/userInput/aiResponse 그대로.
- **끝났다는 기준**: sim-ai·real-ai 화면이 폰 안에서 진행, 타이핑 점 연출, reduced-motion 시 연출 생략, TTS 정상, 콘솔 에러 0, 스크린샷 확인.

## P3 — 미션 엔진 (1회성 개발, 이번 개편의 심장)

만드는 순서 (커밋 단위):

1. **타입 + 컨테이너 뼈대** — `MissionContent` 타입, `MissionStep.tsx`(탭·이름 입력·상태 공유·localStorage), LessonView에 `mission` step kind 연결. 샘플 데이터 1개로 화면 확인.
2. **클릭 블록 4종** — multi-pick, single-pick, summary, vow. (드래그 없이 완성되는 부품 먼저)
3. **드래그 블록 2종 + 그리기** — drag-sort, drag-build(+아이미 응답 분기), draw(캔버스). 클릭 대체 조작 포함.
4. **대화·탐색 + 보상·인쇄** — branch-chat(PhoneFrame 재사용), scene-hunt, 보상 연출, `printMission.ts`(학습지/수료증 인쇄).

- 각 커밋마다: lint·encoding + 브라우저 실사용 확인(스펙 리뷰·품질 리뷰 서브에이전트).
- **끝났다는 기준**: 설계서 §8-3 (블록 9종 전부 동작 + 인쇄 + 복원).

## P4 — 단원 1 파일럿 (미션 11개)

- **작업자에게 주는 것**: `docs/worker-guides/mission-authoring-guide.md` (블록 사용법 + 단원 1 차시별 미션 아이디어 확정본 + 톤 규칙).
- **하는 일**: ① `src/data/lessons/m1.ts`에 mission 스텝 추가 ② 겹치는 기존 퀴즈 스텝 축소 ③ 미션 콘텐츠 작성.
- **커밋 단위**: 3~4차시씩 묶어 2~3커밋.
- **검증**: 리뷰 + 브라우저에서 11개 미션 전부 실제 플레이(드래그·인쇄 포함).

## P5 — 사용자 확인 게이트 ⛔

단원 1을 사용자가 직접 써보고 승인해야 P6 진행. (미션의 재미·난이도·인쇄물 품질을 여기서 조정)

## P6 — 단원 2~6 확장 (미션 57개)

- P4와 같은 방식을 단원별로 반복. 지침서의 "단원별 방향 표"를 따라 각 단원 시작 시 차시별 아이디어를 먼저 표로 확정 → 사용자에게 아이디어 표만 빠르게 확인받고 → 콘텐츠 작성.
- 단원당 1~2커밋, 단원마다 브라우저 검증.

## 역할 분담 정리

| 역할 | 담당 | 근거 문서 |
|---|---|---|
| 목표 수정 (반복) | Sonnet 작업자 | goal-rewrite-guide.md |
| 미션 콘텐츠 작성 (반복) | Sonnet 작업자 | mission-authoring-guide.md |
| 엔진·컴포넌트 개발 (1회성) | 서브에이전트 + 스펙/품질 리뷰 2중 | 설계서 §4·5 |
| 아이디어 표·게이트 승인 | 사용자 | 이 계획서 P5·P6 |

## 위험 요소와 대비

| 위험 | 대비 |
|---|---|
| 미션이 재미없거나 너무 어려움 | P5 게이트에서 단원 1만 보고 조정 — 나머지 57개 착수 전 |
| 드래그가 학생에게 어려움 | 모든 드래그 블록에 클릭 대체 조작 필수 (설계서 §5.6) |
| 인쇄 팝업 차단 | 차단 시 안내 문구 + 재시도 버튼 |
| 콘텐츠 품질 편차 (작업자 반복 작업) | 지침서에 체크리스트 포함 + 단원별 리뷰어 검사 후 커밋 |
| 차시 이미지 없는 scene-hunt | 기존 차시 그림(webp) 재사용 원칙, 없으면 그 차시엔 다른 블록 사용 |
