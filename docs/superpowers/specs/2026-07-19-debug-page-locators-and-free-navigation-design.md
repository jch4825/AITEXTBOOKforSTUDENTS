# 디버그 페이지 식별자와 자유 이동 설계

**상태:** 사용자 승인 완료 (2026-07-19)

## 1. 문제

현재 교과서에는 실제 데이터 도구와 연결되지 않은 AI가 가게 가격, 학교 급식, 교실 자리처럼 알 수 없는 최신·개인 정보를 답하는 장면, 번역투 문장, 내용과 맞지 않는 활동, 저품질 자료가 섞여 있다. 이를 차시별로 교정하려면 사용자가 화면의 위치를 길게 설명해야 하며, 개발자는 같은 위치를 다시 찾아야 한다.

또한 핵심 경험 스튜디오는 활동을 완료하지 않으면 하단 `다음` 버튼이 잠긴다. 콘텐츠를 연속 검토할 때 매 화면에서 임시 응답을 만들어야 하므로 감사 속도가 느리고, 임시 응답이 과정기록에 섞일 위험도 있다.

## 2. 목표

1. 모든 차시에서 활동 여부와 관계없이 하단 버튼으로 다음 페이지를 열 수 있게 한다.
2. 마지막 페이지에 도착하는 것과 차시 완료를 분리한다.
3. 디버그 모드에서 현재 화면을 짧고 고유하게 지목할 수 있는 식별자를 표시한다.
4. 건너뛴 활동에 가짜 응답이나 빈 과정기록을 만들지 않는다.
5. 기존 학생 화면은 디버그 식별자를 노출하지 않는다.

## 3. 범위 밖

- 이번 변경에서 개별 차시의 사실 오류, 환각 유도 장면, 번역투 문장, 활동 품질을 직접 교정하지 않는다.
- 내부 저장값, 기존 과정기록 스키마, 차시 ID를 일괄 변경하지 않는다.
- 각 페이지로 바로 이동하는 점프 메뉴나 URL 라우팅은 만들지 않는다.
- 교실 도구, 지원 수준, 활동 자체의 상호작용 방식은 바꾸지 않는다.

## 4. 페이지 식별자

### 4.1 형식

숫자 순서와 의미 키를 함께 표시한다.

```text
{lessonId} · P{current}/{total} · {pageKey} [· S{subCurrent}/{subTotal}]
```

예시:

| 화면 | 디버그 식별자 |
| --- | --- |
| 일반 차시의 짝 맞추기 | `m1-l2 · P03/05 · matching` |
| 일반 차시의 정리 | `m1-l2 · P05/05 · wrap-up` |
| 스튜디오 판단 화면 | `m1-l1 · P05/08 · decision` |
| 스튜디오 이야기 3번째 장면 | `m1-l1 · P01/08 · encounter · S03/04` |
| 단원 마무리 | `m1-l11 · P01/01 · module-close` |
| 목록에 없는 알 수 없는 차시 | `m1-l999 · P01/01 · coming-soon` |

`P` 번호는 사용자가 현재 순서를 즉시 이해하기 위한 값이다. `pageKey`는 페이지가 추가되어 숫자가 달라져도 코드에서 같은 화면 종류를 찾기 위한 의미 키다. 이야기 내부 장면처럼 한 페이지 안에서 콘텐츠가 바뀌는 경우에만 `S` 하위 번호를 붙인다.

### 4.2 노출 조건과 위치

- URL에 `debug=1`이 있을 때만 식별자를 표시한다.
- `App.updateUrl`은 이미 알 수 없는 검색 매개변수를 보존하므로 차시와 목차를 이동해도 `debug=1`을 유지한다.
- 공통 프레임의 하단 진행 표시 영역에 선택 가능한 고정폭 텍스트로 출력한다.
- 디버그 모드가 아니면 기존 `01 / 08` 진행 표시와 학생 화면을 그대로 유지한다.
- 식별자는 읽기 순서에 포함하되 장식처럼 반복 낭독되지 않도록 간결한 `aria-label`을 제공한다.

### 4.3 페이지 키 생성

- 일반 차시: 현재 `LessonStep.kind`를 사용한다. 마지막 화면은 `wrap-up`을 사용한다.
- 핵심 경험 스튜디오: `StudioStage` 값을 그대로 사용한다.
- 이야기 장면: `encounter` 뒤에 `sceneIndex + 1`과 전체 장면 수를 추가한다.
- 단원 마무리와 목록에 없는 알 수 없는 차시 화면: 각각 `module-close`, `coming-soon`을 사용한다.
- 모든 문자열 조립은 공통 포매터 한 곳에서 수행한다.

## 5. 자유로운 페이지 이동

### 5.1 공통 규칙

- 일반 모드와 디버그 모드 모두 하단 `다음` 버튼을 항상 누를 수 있다.
- 첫 페이지의 `이전` 버튼은 현재처럼 비활성화한다.
- 마지막 페이지에서는 버튼 문구를 현재처럼 `다 했습니다!`로 표시한다.
- 활동을 건너뛸 때 선택값, 텍스트, 그림, AI 판단 같은 임시 데이터를 자동 생성하지 않는다.
- 활동 자체가 제공하는 내부 완료 동작과 피드백은 유지한다.

### 5.2 스튜디오 잠금 해제

현재 스튜디오는 두 곳에서 이동을 막는다.

1. `MicroLessonFrame.nextDisabled`
2. `studioReducer`의 `canAdvance` 검사

두 잠금을 모두 탐색 흐름에서 제거한다. `next` 액션은 마지막 단계가 아닌 한 다음 `StudioStage`로 이동한다. 필수 응답의 존재 여부는 이동 조건으로 사용하지 않는다.

비주얼 이야기의 네 장면을 모두 보아야 한다는 `visualNovelLocked`도 이동 조건에서 제거한다. 장면을 보지 않고 이동해도 완료 장면까지 탐색할 수 있어야 한다.

## 6. 완료와 과정기록

### 6.1 완료 시점

- 마지막 페이지에 진입하는 것만으로는 진도를 완료 처리하지 않는다.
- 마지막 페이지에서 `다 했습니다!`를 직접 누를 때만 `markCompleted`를 호출한다.
- 일반 차시와 단원 마무리는 기존의 명시적 마지막 버튼 흐름을 유지한다.
- 스튜디오는 `complete` 단계 진입 시 실행되던 자동 완료 effect를 명시적 `finish` 동작으로 옮긴다.

### 6.2 과정기록 저장

다음 학생 과정 필드 중 하나 이상이 있을 때만 스튜디오 과정기록을 저장한다.

- `firstAttempt`
- `aiDecision` 또는 `finalExpression`
- 비어 있지 않은 `artifactSummary`
- `transferExpression`

모든 필드가 비어 있으면 차시 진도는 완료할 수 있지만 빈 `StudioEvidenceV2` 레코드는 만들지 않는다. 일부 필드만 있으면 존재하는 값만 담은 희소 기록을 저장한다. TTS 사용이나 힌트 열기 같은 지원 모드 기록만으로는 학생 과정기록을 새로 만들지 않는다.

## 7. 구성 요소와 데이터 흐름

### 7.1 공통 디버그 유틸리티

`src/utils/debugMode.ts`

- `isDebugMode(search)`로 `debug=1` 여부를 판정한다.
- 페이지 식별자 타입과 포매터를 제공한다.
- 브라우저 전역 상태나 별도 localStorage를 만들지 않는다.

### 7.2 공통 프레임

`MicroLessonFrame`은 현재 `lessonId`, `currentStep`, `totalSteps`에 더해 `pageKey`와 선택적 하위 장면 정보를 받는다. 디버그 모드이면 포매터 결과를 하단 진행 영역에 표시한다. `nextDisabled`는 제거하여 새 호출부가 다시 이동 잠금을 만들지 못하게 한다.

### 7.3 일반 차시

`LessonView`가 현재 단계의 `LessonStep.kind` 또는 `wrap-up`을 `pageKey`로 전달한다. 목록에 없는 알 수 없는 차시는 `coming-soon`을 전달한다.

### 7.4 스튜디오 차시

`StudioLessonView`가 현재 `StudioStage`를 `pageKey`로 전달한다. `VisualNovelExperience`의 장면 선택은 콜백으로 현재 장면 번호를 상위에 알리고, 스튜디오 프레임이 이를 선택적 하위 장면 정보로 전달한다. 차시나 encounter 단계를 벗어나면 하위 장면 정보는 초기화한다.

### 7.5 단원 마무리

`ModuleCloseLessonView`는 `module-close` 키를 전달하고 입력 여부와 무관하게 마지막 버튼을 누를 수 있게 한다.

## 8. 오류와 호환성

- 알 수 없는 `debug` 값은 일반 모드로 취급하며 `debug=1`만 활성화한다.
- 페이지 키가 누락되면 식별자에는 숫자 부분만 표시하고 화면을 중단하지 않는다.
- 기존 과정기록을 읽는 로직과 `StudioEvidenceV2` 스키마는 유지한다.
- 과거에 저장된 부분 기록은 그대로 표시한다.
- 디버그 모드가 아닐 때는 `data-debug-page-id` 속성도 만들지 않는다. 일반 학생 DOM에서 내부 코드 노출을 최소화한다.

## 9. 검증

1. 소스 계약 검사로 모든 `MicroLessonFrame` 호출부가 적절한 `pageKey`를 전달하는지 확인한다.
2. reducer 검사로 응답이 없는 `first-attempt`, `decision`, `artifact`, `transfer`에서 `next`가 다음 단계로 이동하는지 확인한다.
3. 스튜디오 완료 검사로 `complete` 진입만으로는 진도·과정기록이 저장되지 않는지 확인한다.
4. 응답이 전혀 없는 상태에서 마지막 버튼을 누르면 진도만 완료되고 과정기록은 생기지 않는지 확인한다.
5. 일부 응답이 있는 상태에서는 해당 값만 포함한 과정기록이 저장되는지 확인한다.
6. `?debug=1`에서 일반 차시, 스튜디오, 이야기 장면, 단원 마무리 식별자가 형식에 맞게 보이는지 브라우저로 확인한다.
7. 디버그 매개변수 없이 같은 화면을 열었을 때 내부 식별자가 보이지 않는지 확인한다.
8. 차시를 바꾸어도 URL의 `debug=1`이 유지되는지 확인한다.
9. TypeScript, UTF-8 인코딩, 프로덕션 빌드를 실행한다.

## 10. 완료 기준

- 사용자는 어떤 활동도 하지 않고 모든 페이지를 앞뒤로 탐색할 수 있다.
- 마지막 버튼을 누르기 전에는 차시가 완료되지 않는다.
- 건너뛴 활동 때문에 가짜 응답이나 빈 과정기록이 생기지 않는다.
- 사용자는 `?debug=1`에서 보이는 문자열 하나만 전달해 현재 차시, 페이지 순서, 화면 종류, 이야기 장면을 정확히 지목할 수 있다.
- 일반 학생 화면에는 디버그 식별자가 노출되지 않는다.
