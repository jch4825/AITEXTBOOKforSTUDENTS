# 4-4 레슨 재설계 — AI Studio Build 예시를 짧은 논술문 채점기로 교체

**날짜**: 2026-06-03
**상태**: 디자인 합의 완료, 구현 계획 작성 대기

## 배경과 문제

### 현재 상태

`src/data/tutorialData.ts`의 `l4-4` ("AI Studio Build로 나만의 행정 앱 만들기") 레슨은 다음 흐름을 가르친다:

1. 4-1에서 만든 학교 시스템 프롬프트 + 4-2에서 만든 행정 업무 프롬프트 템플릿 복사
2. AI Studio Build에 붙여넣고 "이 규칙대로 가정통신문을 쉽고 빠르게 뽑아주는 도우미 앱을 만들어줘" 지시
3. → 가정통신문 자동 생성 앱이 완성

### 문제

사용자가 실제로 여러 번 시도했으나 AI Studio Build의 Gemini가 자연어 해석 한계로 가정통신문 같은 복잡한 양식의 앱을 안정적으로 생성하지 못함. 결과적으로 레슨이 가르치는 약속이 현실에서 성립하지 않는 "거짓 정보"가 되는 상황.

다른 도구(예: Antigravity)로는 Gemini로도 구현 가능하나 모듈 4 범위에서 새 도구를 소개하는 것은 선을 넘음.

### 해결 방향

AI Studio Build의 한계 안에서 안정적으로 동작하면서도, Build의 차별점(React 기반 UI를 가진 웹앱 생성 — Claude Projects/Gemini Gems와 다른 점)을 살리고, 4-1·4-2의 메시지(시스템 프롬프트 + 프롬프트 템플릿 = 앱)를 그대로 가져갈 수 있는 더 가벼운 예시로 교체한다.

## 디자인

### 새 예시: 짧은 논술문 채점기

학생의 짧은 글(200~400자)을 입력받아 학년군에 맞는 톤으로 항목별 채점·피드백을 돌려주는 앱.

**왜 이 예시인가**:

- **시스템 프롬프트 가치 명확**: 학년군·평가 톤·맞춤법 엄격도가 결과에 직접 작용
- **프롬프트 템플릿(=루브릭) 가치 명확**: 평가 항목과 척도가 그대로 템플릿이 됨
- **UI 차별점 명확**: 점수 카드·시각적 막대·"잘된 점/보완할 점" 박스 같은 구조화된 출력은 챗봇(Project/Gems) 한 줄 응답으로는 어색하고 Build의 React UI에서 가치를 가짐
- **Build의 안정적 구현 범위**: 입력 → LLM 호출 → 결과 표시라는 단순 패턴이라 자연어 해석 부담 적음
- **보안 부담 낮음**: 학생 이름 익명화 안내만 추가하면 됨
- **반복성**: 수행평가·논술 과제 채점마다 반복 활용 가능

### 앱 구조

**입력**
- 학생 글: **텍스트 직접 입력** + **사진 업로드(OCR)** 둘 다 제공
- 학년군 선택 (드롭다운 또는 라디오): 1-2학년군 / 3-4학년군 / 5-6학년군
- 글 주제 입력 (선택)
- "채점하기" 버튼

**시스템 프롬프트 (4-1의 응용)**: 학년군 + 평가 톤에 따라 동적으로 결정
- 1-2학년군: 받침과 한두 문장 표현 격려, 자기 생각 옮긴 시도 자체 인정
- 3-4학년군: 두세 문장으로 이유 들기 평가, 맞춤법 본격 점검하되 격려 톤 유지
- 5-6학년군: 짧은 단락에서 근거와 표현 평가, 주장-근거 구조 점검

**프롬프트 템플릿 (4-2의 응용 = 루브릭)**
- 평가 항목 체크박스: 주제 명확성 / 근거·이유 / 표현·문장 / 맞춤법 / 창의성
- 점수 척도: 3점 또는 5점

**출력 UI**
- 항목별 점수 카드 (시각적 막대 또는 별점)
- 각 항목별 한 줄 피드백
- "잘된 점" / "보완할 점" 두 박스로 구조화
- 종합 한 단락 (학년군 톤에 맞춘 격려)

### 레슨 내 체험 영역 = 프롬프트 빌더 (핵심 변경)

기존 `interactive` 필드는 단순 prompt/answer 구조였으나, 4-4의 메시지를 가장 강하게 체화시키기 위해 **인터랙티브 영역을 "프롬프트 빌더"로 재설계**한다. 4-1·4-2가 행정 중심이라 그 산출물을 그대로 가져다 쓰면 채점기에 부적합한 정보가 많기 때문에, 사용자가 이 레슨 안에서 채점기용 프롬프트를 직접 조립해 보고 그것을 Build에 붙여넣는 흐름을 만든다.

**UI는 두 영역으로 시각적으로 분리한다** — 학습자가 "지금 시스템 프롬프트를 만들고 있구나" / "지금 템플릿을 만들고 있구나"를 인지하면서 조작하도록.

#### 영역 ①: 시스템 프롬프트 빌더 (4-1 응용 — "AI에게 누구로 행동할지")

| 입력 | 선택지 |
|---|---|
| 학년군 라디오 | 1-2학년군 / 3-4학년군 / 5-6학년군 |
| 평가 톤 라디오 | 격려 중심 / 균형 / 엄격 |

#### 영역 ②: 프롬프트 템플릿 빌더 (4-2 응용 — "매번 채워 넣을 양식")

| 입력 | 선택지 |
|---|---|
| 평가 항목 체크박스 | 주제 명확성 / 근거·이유 / 표현·문장 / 맞춤법 / 창의성 |
| 점수 척도 라디오 | 3점 척도 / 5점 척도 |

#### 결과 영역 (두 가지가 조립된 한 덩어리)

선택값이 바뀔 때마다 라이브로 갱신되는 텍스트 영역:

```
[시스템 프롬프트]
당신은 초등 3-4학년군 담임교사로, 균형 잡힌 톤으로...
(학년군과 톤에 따라 동적으로 변함)

[프롬프트 템플릿 — 학생 글 채점]
다음 학생 글을 4개 항목으로 5점 척도 채점:
1. 주제 명확성  2. 근거·이유  3. 표현·문장  4. 맞춤법
출력: 항목별 점수, 한 줄 피드백, 종합 코멘트

[입력 처리]
사진 입력의 경우, OCR로 읽은 텍스트를 사용자에게 먼저 보여주고
확인받은 뒤 채점하시오.
```

- **[전체 복사하기]** 버튼
- **안내 박스**: "이 프롬프트를 AI Studio Build에 붙여넣고 '이 규칙으로 채점하는 앱을 React로 만들어줘. 학생 글은 사진 업로드 또는 텍스트 입력으로 받기'라고 입력하세요. OCR 인식 한계도 프롬프트에 명시되어 있어, 앱이 학생 글을 한 번 보여준 뒤 채점하도록 합니다."

### 본문 흐름 변경 범위

기존 레슨 구조를 최소한으로 변경:

- **교체**: 가정통신문 예시(현 line 1704) → 짧은 논술문 채점기 예시
- **유지**: 수업용 게임 예시(현 line 1705) — Build가 비교적 안정적으로 다룰 수 있는 영역이라 그대로 둠
- **유지**: PRD/바이브코딩 단락(현 line 1712)
- **유지**: '앱과 에이전트의 차이' 박스(현 line 1714)
- **유지**: tip(현 line 1718)

### 정직성 안내 (OCR 한계 경고)

모듈 4 팝업의 "한계 솔직히 명시" 톤과 일관되게, OCR 한계 경고를 다음 세 곳에 분산 배치한다:

1. **레슨 본문**: 채점기 앱 입력 설명 단락에 "사진 업로드는 OCR로 글자를 읽지만, 손글씨나 흐릿한 사진은 잘못 인식될 수 있으므로 채점 결과 전에 인식된 텍스트를 한 번 확인하세요" 한 문장
2. **인터랙티브 빌더 결과 영역의 안내 박스**: "OCR 인식 한계도 프롬프트에 명시되어 있어, 앱이 학생 글을 한 번 보여준 뒤 채점하도록 합니다"
3. **생성되는 프롬프트 자체**: "사진 입력의 경우, OCR로 읽은 텍스트를 사용자에게 먼저 보여주고 확인받은 뒤 채점하시오" — Build가 만드는 앱이 이 UX를 따르도록 유도

### 학생 글 익명화 안내

채점기 앱 입력 설명 단락에 한 문장: "학생 이름이 글에 포함되어 있다면 [학생A] 같은 자리표시자로 익명화한 뒤 입력하세요."

## 구현 영역

다음 파일들이 변경된다:

- **`src/data/tutorialData.ts`** — l4-4 `content` 필드의 가정통신문 예시 단락을 채점기 예시로 교체. `interactive` 필드는 타입이 요구하는 prompt/initialInput/answer를 유지하되, 새 컴포넌트의 진입 안내 문구로 채워서 의미 없는 빈 값 대신 학습자에게 보이는 안내가 되도록 한다. (예: `prompt: "프롬프트 빌더로 채점기 앱용 시스템 프롬프트+템플릿 조립해 보기"`)
- **`src/views/Module4Components.tsx`** — 새 React 컴포넌트 `EssayGraderPromptBuilder` 추가:
  - 영역 ①·②의 상태 관리 (`useState`)
  - 학년군별 시스템 프롬프트 텍스트 매핑 (객체)
  - 항목·척도 → 템플릿 텍스트 동적 생성
  - 라이브 갱신 텍스트 영역
  - 복사 버튼 (clipboard API)
  - 모듈 4 색상 테마(`theme.accent` from `moduleThemes.ts`)와 일관된 시각 디자인
  - 두 영역의 시각적 분리: 각 영역을 border 카드로 감싸고 상단에 "① 시스템 프롬프트 빌더 (4-1 응용)" / "② 프롬프트 템플릿 빌더 (4-2 응용)" 라벨 헤더를 두며, 카드 배경에 옅은 색상 차이를 주어 한눈에 구분되도록 한다.
- **`src/views/Tutorial.tsx`** — m4 인터랙티브 영역 라우팅에 l4-4 → `EssayGraderPromptBuilder` 매핑 추가. (현재 m4의 라우팅 패턴 따라)

## 검증 기준

- 빌더에서 라디오·체크박스를 바꾸면 결과 영역 텍스트가 라이브로 갱신된다.
- 학년군 라디오 변경 시 시스템 프롬프트 단락이 학년군에 맞게 바뀐다.
- 평가 항목을 0개 체크하면 안내 메시지(예: "최소 1개 항목을 선택하세요") 표시.
- 복사 버튼 클릭 시 전체 프롬프트가 클립보드에 들어가고 버튼 라벨이 잠시 "복사됨"으로 바뀐다.
- 모바일 화면(320~480px 가정)에서 두 영역이 세로로 자연스럽게 쌓이고, 결과 영역 텍스트가 가독성 있게 표시된다.
- `npm run lint`와 `npm run check:encoding` 통과.
