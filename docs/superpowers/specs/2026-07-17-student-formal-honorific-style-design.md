# 학생 화면 격식 높임말 통일 설계

## 1. 목적

발달장애 학생을 대상으로 하는 교과서의 모든 학생 노출 문장을 존중의 의미가 분명한 격식 높임말로 통일합니다. 현재 널리 사용된 `~요` 계열 해요체와 일부 반말 표현을 `하십시오체` 중심의 `~합니다`, `~습니다`, `~입니다`, `~합니까?`, `~습니까?`, `~입니까?`, `~해 주십시오` 형태로 바꿉니다.

인사말은 예외로 둡니다. `안녕하세요`처럼 인사 기능을 하는 기존 표현은 바꾸지 않습니다.

## 2. 적용 범위

학생이 실행 화면에서 읽거나 TTS로 듣는 모든 문구에 적용합니다.

- 6개 단원 68차시의 제목, 학습목표, 본문, 정리 문장, 활동 지시, 선택지와 피드백
- 쉬운 설명, 기본 설명, 도전적 설명, 용어 정의와 예시
- 사회상황이야기, 캐릭터 대화, AI 응답과 질문 예시
- 핵심 경험 스튜디오, 일반화 활동, 단원 포트폴리오
- 학생용 사전, AAC·PECS 문구, 게임 안내와 결과 피드백
- 학생 화면의 내비게이션, 버튼, 오류·빈 상태·안전 안내
- 위 문구를 읽는 TTS 텍스트와 AI 대체 응답

다음 항목은 제외합니다.

- 교사 전용 화면의 문구
- 개발 문서, 과거 설계 문서, 코드 주석
- 파일명, 식별자, 경로, 저장 키, API 필드와 같은 비노출 기술 문자열
- 인사 기능을 하는 문구

## 3. 문체 규칙

### 3.1 설명과 서술

- `AI가 답을 만들어요.` → `AI가 답을 만듭니다.`
- `이것은 인공지능이에요.` → `이것은 인공지능입니다.`
- `AI도 틀릴 수 있어요.` → `AI도 틀릴 수 있습니다.`
- `맞아요.` → `맞습니다.`

### 3.2 질문

질문의 의미와 서술어에 맞는 격식 의문형을 사용합니다.

- `무엇일까요?` → `무엇입니까?`
- `AI가 있나요?` → `AI가 있습니까?`
- `확인했나요?` → `확인했습니까?`
- `어떻게 해야 할까요?` → `어떻게 해야 합니까?`

`~까요?`를 모두 `~입니까?`로 기계적으로 바꾸지 않습니다. 명사 서술문은 `~입니까?`, 동작·상태 질문은 `~합니까?` 또는 `~습니까?`를 사용합니다.

### 3.3 지시, 부탁과 권유

- `골라 보세요.` → `골라 보십시오.`
- `말해 주세요.` → `말해 주십시오.`
- `다시 해봐요.` → `다시 해 보십시오.`
- `함께 알아볼까요?` → 문맥에 따라 `함께 알아봅시다.` 또는 `알아보겠습니까?`

활동을 시작하도록 이끄는 문장은 강압적인 명령보다 공동 수행을 나타내는 `~해 봅시다`를 우선 사용합니다. 학생이 직접 조작해야 하는 구체적 지시는 `~해 보십시오`를 사용합니다.

### 3.4 대화, 인용과 예시

학생에게 보이는 캐릭터 대화, AI 응답, 직접 인용된 프롬프트 예시도 같은 문체를 적용합니다. 따옴표 안이라는 이유로 해요체나 반말을 남기지 않습니다.

다만 인사 기능을 하는 `안녕하세요` 등의 표현은 그대로 둡니다. 단어·문법 학습을 위해 특정 표현 자체를 표제로 제시하는 경우에는 원형을 보존하되, 그 표현을 설명하는 주변 문장은 격식체로 바꿉니다.

### 3.5 문장이 아닌 짧은 표지

`오늘의 목표`, `다음`, `이전`, `오늘의 용어`처럼 문장 종결형이 아닌 명사형 UI 표지는 그대로 둡니다. 고유명사, 캐릭터 이름, 단원 번호와 기술 용어도 변경하지 않습니다.

## 4. 구현 원칙

런타임에서 문장을 자동 변환하지 않습니다. 자동 변환은 한국어의 불규칙 활용, 조사, 높임 표현과 문맥을 안정적으로 처리할 수 없으며 TTS 문장도 부자연스럽게 만들 수 있습니다.

대신 다음 절차를 사용합니다.

1. 학생 노출 소스에서 해요체와 비격식 종결 후보를 전수 추출합니다.
2. 반복 표현은 변환 후보를 만들되 최종 결과는 문장별로 검토합니다.
3. 검토된 격식체 문구를 원본 데이터와 컴포넌트에 저장합니다.
4. 정적 계약 검사가 남은 비격식 종결형을 찾아냅니다.
5. 전체 빌드와 실제 브라우저에서 대표 차시를 확인합니다.

기계적 일괄 치환 결과를 검수 없이 저장하지 않습니다.

## 5. 검사 설계

새 문체 계약 검사는 교사 전용 화면과 개발 문서를 제외한 학생 노출 소스를 검사합니다.

- 문장 끝의 `~요`, `~세요`, `~줘`, `~해`, `~야?` 등 비격식 종결 후보를 보고합니다.
- 정확한 인사말만 명시적 예외 목록으로 허용합니다.
- `필요`, `주요`처럼 어미가 아니라 명사인 표현은 문맥 또는 정확한 토큰 기준으로 제외합니다.
- 학생용 데이터와 학생 UI에 예외가 새로 추가되면 이유 없이 허용 목록을 늘리지 않습니다.
- 기존 68차시, 18개 스튜디오, 44개 지원 차시, 6개 포트폴리오 계약을 함께 실행합니다.

## 6. 검수 기준

완료된 결과는 다음 조건을 모두 만족해야 합니다.

1. 학생 노출 문장에 인사말을 제외한 해요체와 반말 종결형이 남아 있지 않습니다.
2. 질문은 서술어에 따라 `입니까?`, `합니까?`, `습니까?`를 문법적으로 구분합니다.
3. 지시는 존중을 유지하면서도 학생이 해야 할 행동을 분명히 전달합니다.
4. 제목, 본문, 선택지, 정답 피드백, 오류 안내와 TTS가 같은 문체 원칙을 사용합니다.
5. 인용문과 캐릭터 대화도 학생 화면의 전체 문체와 충돌하지 않습니다.
6. 인사말은 기존 표현을 유지합니다.
7. 교사 전용 화면, 기술 식별자와 과거 설계 문서는 불필요하게 수정하지 않습니다.
8. TypeScript, UTF-8, 콘텐츠 계약, 프로덕션 빌드가 모두 통과합니다.
9. 일반 차시, 게임, 스튜디오, 사전, 오류 안내를 브라우저에서 대표 표본으로 확인합니다.

## 7. 비목표

- 학습 내용, 차시 순서, 활동 정답이나 난이도 체계를 변경하지 않습니다.
- 문체 통일을 이유로 문장을 새로 추가하거나 설명 분량을 늘리지 않습니다.
- 격식체와 무관한 디자인, 레이아웃, 색상과 기능을 수정하지 않습니다.
