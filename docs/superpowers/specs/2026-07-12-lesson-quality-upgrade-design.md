## 1. 왜 하는가 (진단)

| # | 문제 | 확인된 현황 |
|---|---|---|
| 1 | **학습목표가 허들** | 기본(hard) 목표가 성취기준체("~할 수 있다") + 어려운 용어(예측·원리·종합). 게다가 목표가 약속한 행동(설명하기 등)을 정작 활동에서 해볼 기회가 없음 |
| 2 | **활동이 단조로움** | 전 68차시 활동 190여 개 중 OX 51 + 카드고르기 54 + 짝맞추기 26 = **69%가 같은 3종 반복**. 드래그·만들기·결과물 활동 0개. 활동 하나에 들이는 시간이 너무 짧음 |
| 3 | **모의 대화가 비현실적** | sim-ai 38회 전부 "버튼 누르면 말풍선" 방식. 스마트폰 화면 없음 → 실제 AI를 쓰는 경험과 연결이 안 됨 |

## 2. 확정 결정 (사용자 승인)

1. **3단계 순차 진행**: ① 목표 수정(콘텐츠만) → ② 폰 프레임 → ③ 미션 활동 도입(엔진 개발 + 단원 1 파일럿 → 확장).
2. **학생 목표는 전부 쉬운 행동문**: "오늘은 ~해 봐요 / ~을 만들어 봐요". 교사용 성취기준체는 기존 `objective` 필드(교사 화면 전용)에 이미 있으므로 학생 화면에서 성취기준체 제거. **그 차시 활동에서 실제로 해보는 것만 목표로 약속**한다.
3. **미션(학습지) 컨테이너 도입**: 하나의 활동 안에 2~3개 장(흐름)이 이어지고, 앞 장의 선택이 뒤 장에 모이며, 마지막에 보상(인쇄 가능한 결과물 + 화면 보상)이 나오는 구조. 부품(블록) 9종을 조립해 만든다.
4. **기존 4종 게임(OX·카드·짝맞추기·순서)은 "가벼운 확인용"으로 유지**하되 차시당 반복 수를 줄인다. 미션이 주력.
5. **반복 작업은 Sonnet급 작업자가 지침서(md)를 보고 수행**한다. 엔진·컴포넌트 개발(1회성)만 상세 플랜 기반으로 진행.

## 3. ① 학습목표 수정 — 규칙 요약

대상: `src/data/lessons/hard/m1.ts ~ m6.ts`의 전 68차시 `goal { easy, normal, hard }` (204개 문장).

- **hard(기본)**: "오늘은 ~해 봐요" 행동문 1~2개. 그 차시에서 가르치는 핵심 용어는 1개까지 그대로 쓸 수 있음(어차피 그 차시에서 배우니까). 나머지는 풀어쓴다.
- **normal(기초)**: 행동 1개, 더 짧게.
- **easy(쉬움)**: 가장 짧은 한 문장.
- **금지**: "~할 수 있다", 두 겹 이상의 종속절, 활동에 없는 행동 약속.
- 상세 규칙·예시·체크리스트는 `goal-rewrite-guide.md` 참조.

예시 (m1-l3):
- (전) "생성형 AI가 학습한 자료를 바탕으로 다음 말을 예측해 답을 만드는 원리를 설명할 수 있다."
- (후) "오늘은 AI가 다음 말을 이어 붙여 답을 만드는 모습을 구경하고, 내 말로 말해 봐요."

## 4. ② 폰 프레임 (PhoneFrame)

새 컴포넌트 `src/components/PhoneFrame.tsx`:

- 스마트폰 외형(둥근 모서리, 상단 상태바(시간·배터리), 하단 홈바) 안에 채팅 화면을 담는다.
- 채팅 화면: 상단 "아이미" 이름표(아바타), 말풍선 목록(학생=오른쪽, 아이미=왼쪽), 하단 입력창+전송 버튼.
- **타이핑 연출**: 전송을 누르면 학생 말풍선이 뜨고 → 아이미 쪽에 점 3개(···) 타이핑 표시 → 잠시 후 답 말풍선. `prefers-reduced-motion`이면 연출 생략하고 바로 표시.
- 적용 대상: 기존 **sim-ai 38회 전부**(renderSimAI 재작성) + **real-ai 12회**(RealAIStep 내부를 폰 프레임으로) + ③의 분기 대화 블록. 세 곳이 같은 컴포넌트를 쓴다.
- 콘텐츠(prompt/userInput/aiResponse)는 그대로 재사용 — **데이터 수정 없음, 표현만 교체**.

## 5. ③ 미션(학습지) 아키텍처

### 5.1 구조

```
미션 1개 = 표지(제목·이름 입력) + 장(챕터) 2~3개 + 보상
├─ 각 장 = 블록 1~2개 (아래 9종에서 조립)
├─ 장 사이 상태 공유: 앞 장에서 고른 것을 뒤 장 요약표가 자동으로 모음
└─ 보상 = 🖨️ 인쇄물(A4 학습지 또는 수료증/배지판) + 화면 도장·폭죽
```

- 차시 안에서의 위치: **정리(wrap-up) 직전 마지막 활동 스텝**. 새 step kind `mission`.
- 미션 완료 여부와 무관하게 하단 "다음" 버튼으로 항상 진행 가능(학생이 막히지 않게).
- 진행 상태는 `localStorage("ai-students-mission-{lessonId}")`에 저장(선택·입력만, 그림은 제외) — 실수로 나가도 이어서 가능.

### 5.2 타입 스키마 (`src/types.ts`에 추가)

```ts
export interface MissionContent {
  title: string;              // 학습지 제목 (예: "AI 찾기 탐험대")
  intro?: string;             // 시작 안내 한 줄 (자동 TTS)
  askName?: boolean;          // 이름 입력 바 표시 (기본 true)
  chapters: MissionChapter[]; // 2~3개
  reward: MissionReward;
}
export interface MissionChapter {
  title: string;              // 탭 이름 (예: "1장 찾아보기")
  goal?: string;              // 이 장에서 하는 일 한 줄
  blocks: MissionBlock[];     // 1~2개
}
export interface MissionReward {
  printable: 'worksheet' | 'certificate'; // 학습지 / 수료증·배지판
  badgeLabel: string;         // 화면 보상 문구 (예: "AI 탐험가 배지 획득!")
}
```

### 5.3 블록(부품) 9종

| kind | 이름 | 하는 일 | 조작 |
|---|---|---|---|
| `multi-pick` | 복수선택 카드 | "해당하는 것을 모두 골라요" — 이모지/아이콘+라벨 카드 그리드 | 클릭 |
| `single-pick` | 단일선택 카드 | "하나만 골라요" | 클릭 |
| `drag-sort` | 드래그 분류함 | 카드를 2~3개 바구니로 끌어다 분류. 오답은 제자리로 | **드래그** (클릭 대체 조작도 제공: 카드 클릭→바구니 클릭) |
| `drag-build` | 드래그 조립 | 조각(무엇을/어떤 상황/어떤 모양)을 슬롯에 끼워 문장(프롬프트) 조립 → 조립 품질에 따라 아이미 답이 달라짐 | **드래그** (클릭 대체 조작 제공) |
| `branch-chat` | 분기 대화 | 폰 프레임 속 3~5턴 대화, 매 턴 선택지 2~3개 → 선택에 따라 대화가 갈라짐 | 클릭 |
| `scene-hunt` | 장면 탐색 | 큰 그림에서 목표물을 모두 찾아 터치 → 찾으면 스티커 | 클릭 |
| `draw` | 그리기 캔버스 | 자유 그리기(색 6개·굵기 2개·지우기). 인쇄물에 그림 포함 | 드로잉 |
| `summary` | 자동 요약표 | 앞 블록들의 선택 결과를 계획표/모음표로 자동 표시 (블록 id 참조) | 없음(표시) |
| `vow` | 다짐 문장 | "나 {이름}는 꼭 {빈칸} 하겠습니다" 빈칸 채우기 + TTS로 읽기 | 입력 |

각 블록의 데이터 필드 상세는 `mission-authoring-guide.md`(작업자용)에 예시와 함께 정의한다.

### 5.4 컨테이너가 공통 제공 (블록이 신경 안 써도 되는 것)

- 이름 입력·유지, 장 탭 이동, 장별 자동 TTS 안내, 진행률 점, 상태 공유(블록 id → 결과 맵), 인쇄 뷰 생성, 보상 연출, localStorage 저장/복원.
- **인쇄**: 사용자 예시 학습지와 같은 방식 — 선택 결과·다짐·그림(캔버스→이미지 변환)을 취합한 A4용 HTML을 새 창으로 열어 인쇄/PDF. 팝업 차단 시 안내 문구. 수료증(`certificate`)은 이름+단원명+배지 그림이 들어간 상장형 템플릿.

### 5.5 새 파일 (예상)

```
src/components/mission/
├── MissionStep.tsx        — 컨테이너 (탭·이름·상태·보상·인쇄 버튼)
├── printMission.ts        — 인쇄용 HTML 생성 (worksheet/certificate 템플릿)
├── blocks/
│   ├── MultiPick.tsx  SinglePick.tsx  DragSort.tsx  DragBuild.tsx
│   ├── BranchChat.tsx  SceneHunt.tsx  DrawPad.tsx  SummaryTable.tsx  Vow.tsx
└── (공용) useMissionState.ts — 블록 결과 맵 + localStorage
src/components/PhoneFrame.tsx — ②에서 개발, branch-chat이 재사용
```

### 5.6 드래그 구현 원칙

- 외부 라이브러리 없이 **포인터 이벤트 기반**(마우스+터치 공용).
- 모든 드래그 블록은 **클릭 대체 조작**을 함께 제공(카드 클릭 → 목적지 클릭) — 소근육 조작이 어려운 학생 대응.
- 드롭 성공=체크 배지+칩 애니 1회, 실패=흔들림 1회 후 제자리. `prefers-reduced-motion` 대응.

## 6. 차시 재구성 원칙 (③ 콘텐츠 작업)

- 차시당 **미션 1개**(마무리 차시 l11/l12는 2개까지 허용).
- 기존 퀴즈 스텝(OX·카드·짝맞추기)은 차시당 **1~2개로 축소** — 미션과 겹치는 내용의 퀴즈부터 제거.
- 미션의 장 구성은 반드시 그 차시의 (수정된) 목표와 일치 — 목표가 약속한 행동을 미션에서 실제로 한다.
- 단원별 미션 아이디어와 작성 절차는 `mission-authoring-guide.md` 참조 (단원 1은 11차시 전부 아이디어 확정, 단원 2~6은 방향 표 제공 후 각 단계에서 확정).

## 7. 접근성·톤 (불변 원칙)

- 정오 피드백은 색+아이콘+TTS 3중. 색만으로 판단하게 두지 않는다.
- 애니메이션은 생동감 있게 쓰되 `prefers-reduced-motion` 사용자는 연출 생략.
- 터치 타깃 64px(핵심 조작), 포커스 링 유지.
- 문장은 존댓말, 한 문장 한 정보. AI 만능 묘사 금지(기존 콘텐츠 규칙 유지).
- 학생을 막지 않는다: 미션을 끝내지 않아도 차시 진행 가능, 오답에 항상 재시도.

## 8. 완료 기준

1. ① 68차시 목표 전부 행동문 + 활동 일치 (지침서 체크리스트 통과).
2. ② sim-ai·real-ai 50회 전부 폰 프레임 렌더, 타이핑 연출, reduced-motion 대응.
3. ③ 블록 9종 + 컨테이너 동작, 단원 1 미션 11개에서: 장 간 상태 공유·인쇄물 생성·보상 연출·localStorage 복원이 실제로 동작.
4. `npm run lint` + `check:encoding` + `build` 통과, 브라우저 실사용 검증(드래그·인쇄 포함), 콘솔 에러 0.
5. 단원 1 파일럿을 사용자가 확인·승인한 뒤에만 단원 2~6 확장.

## 9. 하지 않는 것 (범위 제외)

- 서버 저장·로그인·학생 계정 (localStorage만).
- 효과음·배경음악 (TTS만).
- 기존 4종 게임의 추가 개편 (유지·축소만).
- 교사도구 '학습지' 버튼과의 연동 (추후 과제로 메모만).
- 태블릿 미만 소형 모바일 최적화 (PC 우선 원칙 유지 — 드래그는 태블릿까지 동작 확인).
