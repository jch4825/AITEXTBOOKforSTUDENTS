# 1·2단원 경험 중심 스튜디오 확장 설계

## 1. 목적과 범위

5단원 파일럿에서 검증한 경험 우선 학습 문법을 1단원과 2단원에 순차 확장한다. 기존 68차시, 차시 번호와 제목, 진도 데이터, 3수준 설명, 이전 일반화 기록은 유지한다. 새 화면은 1·2단원의 핵심 경험 6개와 단원 마무리 2개에만 적용하고, 지원 차시 14개에는 기존 본문 앞의 연결 카드만 추가한다.

이번 단계의 AI 결과는 모두 `prepared` 출처를 사용한다. 카메라, 파일 업로드, 실제 음성 분석, 외부 AI 연결을 요구하지 않는다. 학생은 네트워크와 API 키가 없어도 모든 활동을 완주할 수 있다. 실제 멀티모달 AI 연결은 후속 단계에서 같은 스튜디오 정의를 확장하는 방식으로 추가한다.

## 2. 교육과정 구성 원칙

- 설명을 읽기 전에 생활 장면과 첫 판단을 만난다.
- 첫 판단 뒤에 기존 3수준 설명을 `생각 도구`로 제공한다.
- 조건 변화는 학생이 방법을 유지하거나 수정할 실제 이유가 되어야 한다.
- AI는 정답지가 아니라 확인 질문자, 대안 제시자, 반론자로 사용한다.
- 학생은 AI 의견을 수용, 수정, 거절할 수 있다.
- 결과물은 다음 장면에서도 다시 사용할 수 있는 카드나 기록으로 만든다.
- 정답률보다 첫 시도, 정보 찾기, AI 비교, 조건 조정, 전이를 평가한다.
- 선택, AAC, 글, 말, 그림 중 차시에 적절한 통로를 제공한다.

## 3. 확장 순서

구현과 검증은 단원 완결형으로 진행한다.

1. 공통 데이터 구조를 1·2·5단원에 사용할 수 있게 일반화한다.
2. 1단원의 스튜디오 3개, 지원 브리지 7개, 포트폴리오를 완성하고 검증한다.
3. 2단원의 스튜디오 3개, 지원 브리지 7개, 포트폴리오를 완성하고 검증한다.
4. 교사 기록과 포트폴리오에서 1·2·5단원 증거를 함께 확인한다.
5. 기존 차시와 마지막 정리 화면의 TTS 회귀를 검증한다.

한꺼번에 여섯 스튜디오를 노출하지 않는다. 1단원 완결 흐름이 통과한 뒤 2단원 레지스트리를 공개한다.

## 4. 1단원 교육과정 흐름

1단원은 AI의 정의를 먼저 암기하는 단원에서 생활 사례를 관찰하고 AI의 능력과 한계를 근거로 판단하는 단원으로 바꾼다.

| 구간 | 역할 | 핵심 내용 | 결과물 |
|---|---|---|---|
| 1차시 | 스튜디오 | 오늘 하루의 AI 찾기 | 나의 AI 발견 카드 |
| 2~3차시 | 지원 | 기계와 AI의 차이, AI가 답을 만드는 방법 | 1차시 회상, 4차시 예고 |
| 4차시 | 스튜디오 | AI의 눈과 귀 실험실 | AI 인식 실험 기록 |
| 5~9차시 | 지원 | 음성 인식, 학습 자료, 잘하는 일과 못하는 일, AI 종류 | 4차시 회상, 10차시 예고 |
| 10차시 | 스튜디오 | AI 능력 시험소 | AI 사용 판단 설명서 |
| 11차시 | 단원 마무리 | 세 경험 비교와 새 장면 전이 | 1단원 성장 포트폴리오 |

### 4.1 1차시: 오늘 하루의 AI 찾기

- **생활 장면:** 아침부터 학교에 도착하기까지 사용하는 여러 물건을 살펴본다.
- **첫 판단:** 어떤 물건이 AI를 사용할 가능성이 있는지 선택, AAC, 글, 말, 그림으로 표현한다.
- **조건 변화:** 인터넷이 끊기거나, 같은 물건이 정해진 동작만 하거나, 추천 결과가 사용자마다 달라지는 조건을 추가한다.
- **준비된 AI 관점:** 겉모습이 아니라 입력을 바탕으로 결과가 달라지는지 확인하도록 질문한다.
- **판단:** 첫 선택을 유지해도 근거가 타당하면 성공으로 본다.
- **결과물:** 물건 이름, AI라고 판단한 단서, 확인할 점을 담은 `나의 AI 발견 카드`.
- **전이:** 학교 교실과 도서관에서 새 사례를 판단한다.

### 4.2 4차시: AI의 눈과 귀 실험실

- **생활 장면:** 교과서에 준비된 사물 그림과 짧은 소리 문장을 AI가 어떻게 인식했다고 표시하는지 비교한다.
- **첫 판단:** AI가 맞게 알아본 것, 헷갈린 것, 확인이 필요한 것을 고른다.
- **조건 변화:** 그림 일부 가림, 비슷한 모양, 작은 글자, 주변 소음 같은 변수를 추가한다.
- **준비된 AI 관점:** `확실하지 않으면 다시 보여 달라고 하거나 사람이 확인해야 한다`는 확인 질문을 제시한다.
- **결과물:** 입력 조건, AI가 말한 것, 내가 확인한 것을 나누어 쓰는 `AI 인식 실험 기록`.
- **전이:** 흐린 표지판이나 잘 들리지 않는 안내를 만났을 때의 확인 방법을 판단한다.

소리 자극은 학생이 `소리 듣기` 버튼을 눌렀을 때만 브라우저 TTS로 재생한다. 자동 재생하지 않으며 마이크 권한을 요청하지 않는다. 이미지 자극은 기존 교과서·AAC 자산을 재사용하고 얼굴, 이름, 주소가 포함된 사진을 사용하지 않는다.

### 4.3 10차시: AI 능력 시험소

- **생활 장면:** 번역, 추천, 마음 알아맞히기, 안전 판단처럼 성격이 다른 일을 AI에게 맡기는 상황을 비교한다.
- **첫 판단:** AI에게 맡길 일, 결과를 확인할 일, 사람에게 부탁할 일을 구분한다.
- **조건 변화:** 답이 빠르지만 근거가 없거나 개인정보가 필요하거나 실제 안전과 관련된 조건을 추가한다.
- **준비된 AI 관점:** AI가 잘하는 일도 결과 확인이 필요하다는 반론을 제시한다.
- **결과물:** `AI에게 맡기기 / 확인하고 쓰기 / 사람에게 부탁하기`를 정리한 `AI 사용 판단 설명서`.
- **전이:** 처음 보는 AI 기능을 사용할지 판단한다.

## 5. 2단원 교육과정 흐름

2단원은 좋은 프롬프트 규칙을 암기하는 단원에서 내 뜻이 전달되지 않았을 때 요청을 만들고 반복해서 고치는 단원으로 바꾼다.

| 구간 | 역할 | 핵심 내용 | 결과물 |
|---|---|---|---|
| 1차시 | 스튜디오 | AI가 내 말을 못 알아들었어요 | 요청 고치기 카드 |
| 2~5차시 | 지원 | 짧게 묻기, 구체화, 예시, 역할 지정 | 1차시 회상, 6차시 예고 |
| 6차시 | 스튜디오 | 요청 공동 제작소 | 나의 요청 제작 레시피 |
| 7~9차시 | 지원 | 다시 묻기, 길이 조절, 이상한 답 확인 | 6차시 회상, 10차시 예고 |
| 10차시 | 스튜디오 | AI 고쳐 묻기 실험실 | 고쳐 묻기 대화 기록 |
| 11차시 | 단원 마무리 | 세 경험 비교와 새 장면 전이 | 2단원 성장 포트폴리오 |

### 5.1 1차시: AI가 내 말을 못 알아들었어요

- **생활 장면:** 학생이 `그거 알려 줘`처럼 조건이 빠진 요청을 했고 준비된 AI가 다른 뜻으로 응답한다.
- **첫 판단:** 같은 말을 반복할지, 필요한 정보를 더할지, 다른 사람에게 물을지 표현한다.
- **조건 변화:** 원하는 대상, 시간, 답의 형태 중 두 가지가 새로 필요해진다.
- **준비된 AI 관점:** 개인정보가 아닌 확인 질문으로 뜻을 좁힌다.
- **결과물:** 처음 요청과 안전하게 덧붙인 정보를 비교하는 `요청 고치기 카드`.
- **전이:** 이름을 모르는 물건이나 활동을 부탁하는 장면에 적용한다.

### 5.2 6차시: 요청 공동 제작소

- **생활 장면:** 학교 준비를 도와 달라는 복합적인 요청을 만든다.
- **첫 판단:** 선택, AAC, 글, 말, 예시, 단계 카드 중 자신에게 맞는 통로로 시작한다.
- **조건 변화:** 시간이 줄거나, 답을 표로 받아야 하거나, 한 번에 한 단계만 처리해야 하는 조건을 추가한다.
- **준비된 AI 관점:** 학생의 요청을 대신 완성하지 않고 빠진 조건을 묻거나 다른 배열을 제안한다.
- **결과물:** 목적, 필요한 정보, 예시, 순서, 원하는 답의 모양을 정리한 `나의 요청 제작 레시피`.
- **전이:** 다른 학습 또는 생활 부탁에 같은 제작 순서를 적용한다.

### 5.3 10차시: AI 고쳐 묻기 실험실

- **생활 장면:** 너무 길거나, 엉뚱하거나, 확실하지 않은 준비형 AI 답을 만난다.
- **첫 판단:** 답을 그대로 사용할지, 짧게 해 달라고 할지, 근거를 확인할지, 다른 방법을 찾을지 정한다.
- **조건 변화:** 시간 제한, 읽기 수준, 사실 확인 필요성 중 두 가지 이상을 추가한다.
- **준비된 AI 관점:** 첫 수정 요청 뒤에도 완벽하지 않은 두 번째 응답을 제시해 한 번 더 판단하게 한다.
- **결과물:** 처음 요청, 첫 AI 답, 수정 요청, 달라진 답, 최종 판단을 담은 `고쳐 묻기 대화 기록`.
- **전이:** 새로운 AI 답을 받은 뒤 수용, 수정, 거절을 다시 선택한다.

기존 차시 제목 `진짜 AI랑 놀아보기`는 데이터 호환을 위해 유지한다. 새 화면에는 `준비된 AI 예시`와 `안전한 연습 응답`이라는 표시를 지속적으로 보여 실제 AI 응답으로 오해하지 않게 한다.

## 6. 형식적 다양성

여섯 스튜디오가 모두 같은 8단계 상태 전이를 사용하더라도 학생이 하는 일은 다르게 구성한다.

| 스튜디오 | 주요 형식 | 개방형 요소 |
|---|---|---|
| 1-1 | 생활 장면 탐색, 근거 카드 분류 | 학생이 직접 발견한 AI 사례 표현 |
| 1-4 | 이미지 비교, 버튼으로 듣는 소리, 관찰 기록 | AI가 헷갈린 이유 추측 |
| 1-10 | 과제 분류, 반론 비교, 사용 설명서 | 사람과 AI의 역할 경계 설명 |
| 2-1 | 실패한 요청 수선 | 빠진 정보를 학생 방식으로 보완 |
| 2-6 | AAC·글·말·예시·단계 조립 | 요청 레시피 공동 제작 |
| 2-10 | 두 번의 응답 비교와 반복 수정 | AI 의견 수용·수정·거절 및 이유 |

정답 카드만 반복하지 않는다. 이유 입력은 선택 사항으로 두되, 결과물 단계에는 학생의 표현을 그대로 반영한다. 그림 원본은 현재 화면에서만 사용하고 과정기록에는 `그림으로 표현함`이라는 정제된 표시만 저장한다.

## 7. 소프트웨어 구조

### 7.1 스튜디오 레지스트리

- `src/data/studios/m1.ts`: 1단원 스튜디오 3개.
- `src/data/studios/m2.ts`: 2단원 스튜디오 3개.
- `src/data/studios/index.ts`: 완성 정의 9개를 한 조회기에 등록한다.
- 역할 레지스트리의 18/44/6 분류는 변경하지 않는다.

### 7.2 준비형 자극

`StudioDefinition`에 선택적인 준비형 자극을 추가한다.

```ts
type PreparedStimulus =
  | { kind: 'image'; src: string; alt: string; caption: string }
  | { kind: 'speech'; text: string; label: string };
```

자극은 `encounter` 또는 `conditionChange`에 선택적으로 연결한다. 이미지 로딩 실패 시 `alt`와 `caption` 카드가 남는다. 소리는 직접 누른 버튼에서만 `speakNow`를 호출하며 재생하지 않아도 같은 텍스트를 읽고 활동할 수 있다. 자극 원본이나 재생 이력은 과정기록에 저장하지 않는다.

### 7.3 지원 브리지

- `SupportBridgeDefinition`을 M5 파일 밖의 공통 타입으로 이동한다.
- `src/data/supportBridges/m1.ts`, `m2.ts`, `m5.ts`를 공통 레지스트리에서 조회한다.
- 1단원 7개, 2단원 7개, 5단원 8개로 완성 브리지는 22개가 된다.
- 브리지는 지원 차시의 첫 설명 화면에 한 번만 표시하고 기존 설명·게임·미션을 바꾸지 않는다.

### 7.4 단원 마무리 포트폴리오

`ModuleCloseLessonView`의 5단원 하드코딩을 `ModulePortfolioDefinition` 데이터로 분리한다.

```ts
interface ModulePortfolioDefinition {
  lessonId: LessonId;
  moduleId: ModuleId;
  crumb: string;
  kicker: string;
  title: string;
  description: string;
  studioLessonIds: readonly LessonId[];
  nextChoices: StudioChoice[];
}
```

1·2·5단원은 공통 포트폴리오 화면을 사용한다. 기록이 꺼져 있거나 해당 단원 기록이 없으면 활동하지 않았다는 의미가 아니라는 빈 상태를 표시한다. 기록이 없어도 자기평가와 다음 방법 표현을 완료하면 기존 진도는 저장한다. 3·4·6단원 마무리는 정의가 생길 때까지 기존 화면을 유지한다.

### 7.5 라우팅과 하위 호환

라우팅 순서는 다음과 같다.

1. 완성된 스튜디오 정의가 있으면 `StudioLessonView`.
2. 완성된 포트폴리오 정의가 있으면 `ModuleCloseLessonView`.
3. 그 외는 기존 `ImplementedLesson`.

역할이 `studio` 또는 `module-close`라는 이유만으로 새 화면을 열지 않는다. 정의 누락 시 기존 차시가 항상 폴백이 된다.

## 8. 데이터 흐름과 저장

학생의 현재 스튜디오 상태는 기존 `studioReducer`와 `useStudioSession`을 그대로 사용한다. 완료 시 기록 설정을 확인하고, 교사가 과정기록을 켠 경우에만 기존 `StudioEvidenceV2` 형식으로 저장한다.

```text
스튜디오 정의
→ 첫 표현과 조건 변화
→ 준비된 AI 의견
→ 학생의 수용·수정·거절
→ 결과물과 전이
→ 정제된 StudioEvidenceV2
→ 단원 포트폴리오와 교사 기록
```

새 스키마 버전을 만들지 않는다. 기존 5단원 기록, 암호화 백업, 개별 삭제, 전체 삭제, 교사 관찰기록은 그대로 작동해야 한다. 음성 원본, 이미지 원본, 그림 원본, 전체 응답 대화는 저장하지 않는다.

## 9. TTS 동작 통일

현재 기존 차시의 마지막 정리 화면은 `LessonView`의 효과에서 `speak(wrapUpText)`를 자동 호출한다. 이는 새 스튜디오만 수정해서 해결되지 않는 공통 회귀다.

- 기존 마지막 정리 화면의 자동 `speak()` 효과를 제거한다.
- 스튜디오 단계 진입과 포트폴리오 진입에서도 자동 호출하지 않는다.
- 목표, 본문, 준비형 AI 의견, 소리 자극, 정리 내용은 각 읽기 버튼의 직접 조작으로만 재생한다.
- TTS 토글이 켜져 있어도 화면 진입만으로 `speechSynthesis.speaking`이 시작되지 않아야 한다.

## 10. 오류와 안전 처리

- 준비된 이미지가 로드되지 않으면 대체 텍스트와 설명으로 활동을 계속한다.
- 소리 재생을 지원하지 않거나 학생이 사용하지 않으면 화면의 동일 문장으로 활동한다.
- 준비된 AI 응답은 항상 출처 배지로 구별한다.
- 스튜디오 또는 포트폴리오 정의가 누락되면 기존 차시로 폴백한다.
- 과정기록이 꺼져 있거나 저장이 실패해도 학습과 진도 완료는 막지 않는다.
- 학생이 AI 의견을 거절하거나 선택을 바꾸지 않아도 실패 문구를 표시하지 않는다.
- 준비형 이미지에는 얼굴, 학생 작품, 실명, 주소, 연락처, 위치 정보가 들어가지 않는다.
- 이번 단계에서는 카메라, 마이크 입력, 파일 업로드, 외부 네트워크 권한을 새로 요청하지 않는다.

## 11. 검증 전략

### 11.1 정적 계약 검사

- 역할 레지스트리는 18개 스튜디오, 44개 지원, 6개 마무리를 유지한다.
- 완성 스튜디오는 1·2·5단원 각 3개, 합계 9개다.
- 준비된 AI 출처는 여섯 새 스튜디오 모두 `prepared`다.
- 완성 지원 브리지는 1단원 7개, 2단원 7개, 5단원 8개다.
- 완성 포트폴리오는 1·2·5단원 3개다.
- 준비형 자극에는 대체 텍스트 또는 화면 문장이 반드시 존재한다.
- 기존 마지막 정리 화면에 자동 `speak(wrapUpText)` 호출이 없어야 한다.

### 11.2 단계별 자동 검사

1단원 공개 전에 스튜디오·브리지·포트폴리오·TTS 계약, 타입 검사, 프로덕션 빌드를 실행한다. 2단원도 같은 검사를 반복한다. 마지막에는 기존 일반화, UI, AAC 아이콘, 인코딩 검사를 포함한 전체 검사를 실행한다.

### 11.3 브라우저 시나리오

- 1-1을 기록 끔 상태로 완주하고 진도만 남는지 확인한다.
- 1-4에서 이미지 대체 텍스트와 직접 조작 소리 버튼을 확인한다.
- 1-10에서 AI 의견 거절이 실패로 표시되지 않는지 확인한다.
- 1-11에서 세 증거의 첫 생각, 최종 판단, 전이를 비교한다.
- 2-1에서 첫 판단 직후 정답 해설이 나오지 않는지 확인한다.
- 2-6에서 선택·AAC·글 중 서로 다른 표현 통로로 요청을 만든다.
- 2-10에서 두 번의 준비된 응답을 비교하고 수정 또는 거절한다.
- 2-11과 교사 화면에서 2단원 증거 3개를 확인한다.
- 3-1과 6-1은 아직 기존 화면으로 폴백하는지 확인한다.
- 기존 차시와 새 화면 모두 마지막 단계 진입 전후 `speechSynthesis.speaking`이 `false`인지 확인한다.
- 브라우저 오류 로그가 비어 있는지 확인한다.

## 12. 완료 기준

1. 1·2단원 핵심 경험 6개가 공통 8단계 문법으로 작동한다.
2. 여섯 경험의 생활 장면, 조건 변화, AI 역할, 결과물, 전이 장면이 서로 다르다.
3. 지원 차시 14개가 이전 경험 회상과 다음 경험 예고를 제공하면서 기존 본문을 유지한다.
4. 1·2단원 마무리가 정답 퀴즈가 아닌 세 증거 비교와 자기평가로 작동한다.
5. 준비된 이미지·소리·AI 응답은 권한과 네트워크 없이 이용할 수 있다.
6. 과정기록을 켜지 않아도 모든 활동과 진도 완료가 가능하다.
7. 과정기록을 켜면 1·2·5단원 증거가 같은 교사 화면과 백업에 나타난다.
8. 기존 5단원 동작과 기존 기록이 회귀하지 않는다.
9. 3·4·6단원의 미완성 스튜디오와 마무리는 기존 화면을 유지한다.
10. 마지막 화면을 포함해 어떤 차시도 사용자 조작 없이 TTS를 시작하지 않는다.
11. 계약 검사, 타입 검사, 기존 회귀 검사, 인코딩 검사, 프로덕션 빌드가 성공한다.
