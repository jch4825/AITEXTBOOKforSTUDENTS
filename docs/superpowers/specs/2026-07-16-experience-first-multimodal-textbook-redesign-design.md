# 경험 우선 멀티모달 AI 교과서 2세대 개편 설계

**작성일:** 2026-07-16
**상태:** 대화 설계 승인 완료, 구현 전 사용자 문서 검토 대기
**대상:** 발달장애 중학생 이상 학습자를 위한 「인공지능 활용」 디지털 교과서
**배포 전제:** 정적 웹 배포(GitHub Pages), 백엔드 없음, 교사 선택형 외부 AI 연결

## 1. 설계 배경

현재 교과서는 6단원 68차시의 본문, 목표, 성취기준, 미션, 진도 기능을 갖추었다. 최근에는 단원별로 `첫 생각 → 조건 변화 → AI와 비교 → 최종 판단 → 새로운 장면`을 연결하는 일반화 순환도 추가했다.

그러나 최근 개발은 전체 비평 중 **일반화 부족**에 대한 부분 개선이었다. 교과서 전체 경험을 다시 검토하면 다음 문제가 남아 있다.

1. 대부분의 차시가 `설명 → 확인 문제 → 활동 → 미션 → 정리`를 반복한다.
2. 주제는 다양하지만 학생이 수행하는 인지 행동은 고르기·확인하기·분류하기에 집중되어 있다.
3. 68개 차시에서 카드 선택 49개, OX 48개, 분기 대화 44개, 끌어 분류하기 38개에 비해 그리기 11개, 실제 AI 활동 17개로 수렴형 활동의 비중이 높다.
4. 이미지 인식과 이미지 생성은 개념 설명과 모의 대화 중심이며 실제 비주얼 AI 경험이 부족하다.
5. 쉬움·보통·어려움 설명이 경험 앞에 길게 놓일 경우, 경험 우선 수업이 다시 설명 우선 수업으로 되돌아갈 수 있다.
6. 현재 디자인은 이전보다 정돈되었지만, 매거진형 표지·전통적 교과서 본문·아동용 배지와 이모지 언어가 혼재한다.
7. 과정평가 기록은 단원당 1건, 한 브라우저의 학생 1명 기준으로 제한되어 있으며 교사가 저장 원리와 한계를 이해할 안내 구조가 부족하다.

따라서 다음 개편은 활동 블록을 몇 개 더 추가하는 작업이 아니라, 교과서의 기본 학습 문법을 바꾸는 작업으로 정의한다.

```text
기존: 설명 → 확인 → 기능 연습 → 미션 → 정리

개편: 생활 경험 → 첫 시도 → 필요한 도움 → AI와 비교
      → 수용·수정·거절 → 결과물 → 전이 → 과정 증거
```

## 2. 목표

### 2.1 교육과정 목표

- 68차시와 6단원이라는 행정적 외형을 유지한다.
- 18개 핵심 경험 스튜디오를 중심으로 나머지 차시의 역할을 재배치한다.
- 개념을 먼저 설명하지 않고, 생활 경험 뒤에 개념 이름을 붙이는 귀납적 흐름을 강화한다.
- 명시적 교수와 반복 연습을 없애지 않고 핵심 경험의 앞뒤에 배치한다.
- 정답이 하나인 안전 지식과 정답이 하나가 아닌 생활 판단을 구분한다.

### 2.2 AI 교육 목표

- AI를 답변 도구가 아니라 대안 제안자·확인 질문자·비판자·공동 제작자로 경험한다.
- 학생이 AI 의견을 수용·수정·거절할 수 있게 한다.
- 텍스트·음성·AAC를 넘어 이미지 인식과 이미지 생성을 안전하게 경험한다.
- 실제 AI가 연결되지 않아도 동일한 학습 목표를 달성한다.

### 2.3 특수교육 목표

- 모든 학생이 같은 핵심 생활 문제에 참여하되 지원의 강도와 사고 깊이를 조절한다.
- 지원 수준을 학생 능력 등급이 아니라 현재 활동에 필요한 도움으로 본다.
- 학생의 첫 시도, 사용한 도움, AI 판단, 수정, 전이를 과정 증거로 남긴다.
- 원본 사진·음성·영상은 과정평가 기록에 저장하지 않는다.

### 2.4 디자인 목표

- 실제 중등 교과서의 편집 문법을 디지털 상호작용과 결합한다.
- 저자극 접근성을 유지하면서 청소년에게 유치하지 않은 시각 언어를 사용한다.
- 단원별 색상은 구별을 돕는 제한된 포인트로만 사용한다.
- 배지·보상 중심 언어를 학습 증거·결과물·스튜디오 기록 중심 언어로 바꾼다.

## 3. 범위와 비범위

### 3.1 이번 개편에 포함하는 것

- 68차시 역할 재분류
- 18개 핵심 경험 스튜디오

- 6개 단원 마무리 포트폴리오

- 텍스트·음성·AAC·그림·이미지 멀티모달 표현
- 실제 이미지 인식 5회, 이미지 생성 3회에 해당하는 학습 설계
- 선택형 과정기록과 18개 증거 카드
- 교사 운영 안내 허브
- 현대적 교과서 에디토리얼 UI
- 여섯 단원 색상 토큰

- 학생 계정과 다중 학생 명부
- 클라우드 동기화
- 서버 기반 인증과 학생정보시스템
- LMS 연동
- 원본 사진·음성·영상의 장기 보관
- 얼굴 인식 또는 실제 인물과 닮은 이미지 생성
- AI가 자동으로 학생의 능력이나 장애 정도를 진단하는 기능
- AI가 교사 판단 없이 과정평가 점수를 자동 산출하는 기능

## 4. 68차시 교육과정 재편

### 4.1 차시 역할

| 차시 역할 | 수 | 기능 |
|---|---:|---|
| 핵심 경험 스튜디오 | 18 | 실제 생활 장면에서 판단·비교·제작·전이 |
| 지원 차시 | 44 | 스튜디오에 필요한 개념·기능·안전 규칙의 명시적 교수와 반복 연습 |

### 4.3 단원 리듬

모든 단원은 예측 가능한 세 번의 경험 리듬을 사용한다.

```text
첫 스튜디오: 설명 전에 생활 장면을 경험
↓
지원 차시: 첫 경험에서 필요해진 개념과 기능 학습
↓
중간 스튜디오: AI와 함께 표현·비교·수정
↓
지원 차시: 확인·안전·대안·수정 기능 연습
↓
후반 스튜디오: 여러 조건이 있는 종합 과제와 결과물
↓
단원 마무리: 세 증거 카드 비교와 새로운 장면 전이
```

### 4.4 지원 차시의 기본 구조

지원 차시는 기존 콘텐츠를 버리지 않고 다음 순서로 재배치한다.

```text
지난 스튜디오 장면 20~30초 회상
→ 필요한 개념 한 가지 직접 설명
→ 짧은 집중 연습
→ 다음 스튜디오에서 쓸 도구 카드로 저장
```

지원 차시에서도 OX·카드 선택·매칭을 사용할 수 있다. 다만 연속된 두 지원 차시가 동일한 인지 행동만 반복하지 않게 하고, 다음 스튜디오와 연결되는 목적을 화면에 명시한다.

## 5. 3수준 설명의 통합

### 5.1 핵심 원칙

세 수준은 서로 다른 교육과정이 아니다. 모든 학생이 같은 핵심 생활 문제에 참여하고, 다음 요소만 달라진다.

| 요소 | 충분한 지원 | 보통 | 도전적 |
|---|---|---|---|
| 상황 정보 | 조건을 한 개씩 제시 | 조건 두 개 동시 제시 | 모호하거나 충돌하는 조건 세 개 이상 |
| 표현 | 그림·AAC·음성 우선 | 카드·음성·짧은 글 선택 | 열린 답변·비교 설명 |
| 선택 지원 | 2~3개 선택지와 시각 단서 | 선택지와 이유 카드 | 방법을 학생이 생성하거나 열린 입력 |
| AI 역할 | 확인 질문 하나 | 다른 방법 하나 제안 | 반론·빠진 조건·불확실성 제시 |
| 설명 | 한 문장 도움 카드 | 짧은 개념과 예시 | 용어·근거·한계 확장 |
| 평가 | 시도와 핵심 정보 찾기 | 비교하고 이유 표현 | 근거로 AI 의견 평가·수정 |

화면에서는 지원 수준을 `충분한 지원 · 보통 · 도전적`으로 표시한다.

### 5.2 기존 설명 콘텐츠의 재배치

- `bodyEasy`: 경험 중 자동 또는 눈에 잘 보이는 한 문장 도움 카드로 분해한다.
- `bodyNormal`: 경험 중 선택형 개념 카드와 지원 차시의 핵심 설명으로 사용한다.
- 어려움용 `concept`: 경험 후 개념화 또는 확장 읽기로 이동한다.
- 어려움용 `terms`: 경험한 행동에 교육과정 용어를 붙이는 확장 용어 카드로 사용한다.
- 어려움용 `method`: 스튜디오 중 요청 가능한 절차 도움으로 사용한다.
- 어려움용 `limits`: 경험 후 반드시 확인하는 한계·안전 카드로 사용한다.

### 5.3 스튜디오 안의 설명 위치

1. **경험 전:** 시작에 꼭 필요한 장면 안내만 제공한다. 해결 방법은 가르치지 않는다.
2. **경험 중:** 학생이 막히거나 도움을 선택할 때 기존 설명을 짧게 나누어 제공한다.
3. **경험 후:** 학생이 이미 수행한 행동에 개념 이름을 붙이고 한계·안전 정보를 확인한다.

지원 수준을 활동 중 바꾸더라도 첫 시도와 현재 작업은 사라지지 않는다.

## 6. 핵심 스튜디오 공통 학습 문법

18개 스튜디오는 다음 일곱 단계를 공유한다.

1. **생활 장면과 만남:** 익숙하지만 정답이 하나가 아닌 장면을 그림·소리·짧은 글로 제시한다.
2. **첫 시도:** 카드·AAC·글·음성·그림 중 가능한 방법으로 학생 생각을 표현한다.
3. **조건 변화:** 시간·장소·사람·도구 중 2~3개 조건을 바꾼다.
4. **AI와 비교:** AI가 다른 방법, 확인 질문, 빠진 조건 또는 반론을 제시한다.
5. **학생 판단:** AI 의견을 수용·수정·거절한다. 세 선택 사이에 점수 우열을 두지 않는다.
6. **결과물:** 실제 생활에서 쓸 수 있는 행동 카드·AAC 메시지·시각 일정표·포스터·계획표 등을 만든다.
7. **전이:** 유사하지만 다른 장면에 한 번 더 적용하고 과정 증거를 남긴다.

학생이 선택을 바꾸지 않아도 안전하고 타당한 이유가 있으면 성공으로 본다.

## 7. 하이브리드 멀티모달 AI

### 7.1 운영 방식

모든 스튜디오는 멀티모달 표현을 지원하지만, 모든 스튜디오가 실제 외부 AI를 호출하지는 않는다.

- **교과서 연습 모드:** 기본값. 미리 준비한 안전한 AI 결과를 사용한다.
- **연결형 스튜디오:** 교사가 명시적으로 켠 경우에만 실제 AI를 호출한다.

실제 결과와 준비된 결과를 학생에게 명확히 구별해 표시한다.

### 7.2 기본 표현 통로

- 카드 선택
- AAC 조합
- 글 입력
- 음성 입력
- 그림 표현
- 교과서 제공 이미지 선택

한 화면에는 기본 방식 하나만 먼저 보이고 `다른 방법으로 표현하기`에서 나머지 방식을 연다.

### 7.3 실제 이미지 인식 장면

| 스튜디오 | 학습 경험 |
|---|---|
| 1-4 | 사물 사진의 AI 인식 결과와 실제 사물 비교 |
| 3-9 | 사진 설명에서 맞는 것·빠진 것·추측한 것 찾기 |
| 4-10 | 광고 이미지에서 과장 문구와 확인할 정보 찾기 |
| 5-11 | 비인물 재료 사진을 보고 가능한 조리 계획 만들기 |
| 6-1 | 상품 사진에서 종류·가격·필요 정보 비교 |

### 7.4 실제 이미지 생성 장면

| 스튜디오 | 학습 경험 |
|---|---|
| 3-5 | 학생 장면 설명으로 이미지를 만들고 요청과 결과 차이 수정 |
| 5-11 | 조리 순서를 시각 일정표로 만들고 잘못된 순서 수정 |
| 6-11 | 얼굴 사진 없이 취미·강점을 나타내는 상징 이미지나 아바타 생성 |

이미지 생성의 목표는 멋진 그림이 아니라 `요청 → 생성 결과 → 빠진 것·잘못된 것 → 수정 요청`의 비교 과정이다.

### 7.5 이미지 개인정보 안전선

#### 사용할 수 있는 이미지

- 교과서에 포함된 이미지
- 교사가 준비한 사물·상품·음식·풍경 사진
- 얼굴·명찰·주소·학교명·차량번호가 없는 사진
- 학생이 선택했더라도 교사가 확인한 비인물 사진

#### 기본적으로 사용할 수 없는 이미지

- 학생 또는 가족의 얼굴
- 집 내부·주소·학교명·명찰
- 의료 정보가 보이는 사진
- 다른 학생이 나온 사진
- 현재 위치를 추정할 수 있는 사진

카메라는 자동으로 켜지지 않는다. 이미지 전송 전 `미리보기 → 개인정보 확인 → 외부 AI 전송 안내 → 교사 승인 또는 안전 확인`을 거친다.

### 7.6 이미지 생성 안전선

- 초기 입력은 `대상 · 행동 · 장소 · 표현 방식`의 구조화된 조합을 사용한다.
- 실제 학생·교사와 닮은 인물 생성을 허용하지 않는다.
- 폭력·성적 표현·개인정보·특정 학교나 집 위치·조롱과 차별 요청을 제한한다.
- 모든 생성 결과에 `AI가 만든 이미지`라고 표시한다.
- 생성 횟수와 비용을 통제할 수 있게 교사 설정을 둔다.

### 7.7 연결 실패

- 네트워크·키·제공자 오류 시 `연습용 예시 결과`로 즉시 전환한다.
- 준비된 결과를 방금 생성된 실제 AI 결과처럼 표현하지 않는다.
- 학생의 첫 시도와 작업은 유지한다.
- 이미지·음성 대체 경로를 항상 제공한다.

현재의 정적 배포와 브라우저 직접 AI 키 방식은 학교용 강한 보안 경계가 아니다. 첫 구현에서는 교과서 이미지와 교사가 확인한 비인물 이미지만 실제 AI에 사용한다. 학생 자유 이미지 업로드를 정식 운영하려면 별도 안전 중계 시스템과 기관 정책 검토가 선행되어야 한다.

## 8. 과정중심평가와 포트폴리오

### 8.1 기록 원칙

모든 클릭을 기록하지 않고 다음 다섯 순간만 기록한다.

1. 첫 시도
2. 사용한 도움
3. AI와 비교한 결과
4. 최종 판단 또는 결과물
5. 새로운 상황에 적용한 결과

각 스튜디오는 학생 한 명에게 한 장의 증거 카드를 만들며, 전체 과정에서 최대 18장이 쌓인다.

### 8.2 증거 카드 항목

- 스튜디오 ID와 완료 시각
- 첫 시도의 표현 방식과 짧은 요약
- 사용한 지원 방식과 전체 도움 수준
- AI 결과 출처: 실제 AI 또는 연습용 예시
- AI 역할: 대안·확인 질문·비판·요약·제작
- 학생 결정: 수용·수정·거절
- 최종 판단 또는 구조화된 결과물 요약
- 선택적 이유
- 전이 장면의 시도
- 교사 관찰 기록

원본 사진·원본 음성·카메라 영상·전체 AI 대화 로그는 저장하지 않는다.

### 8.3 교사 관찰 기준

현재의 네 기준을 유지한다.

1. 상황의 중요한 정보를 찾았는가
2. 스스로 한 가지 방법을 시도했는가
3. AI 의견을 비교하고 판단했는가
4. 조건이 달라졌을 때 방법을 조정했는가

각 기준은 `관찰 전 · 도움받아 시도 · 스스로 시도`로 기록한다. 전체 도움 수준은 `혼자 수행 · 말·그림 단서 · 선택지 조정 · 함께 수행`으로 별도 기록한다.

### 8.4 안전 활동의 평가

안전 활동은 완전 개방형으로 처리하지 않는다.

```text
안전 경계를 지켰는가
+
그 경계 안에서 자신의 방법을 선택했는가
```

안전하지 않은 선택을 하면 이유 확인, 안전 정보 제공, 재시도, 수정 과정 기록의 순서로 진행한다. 점수를 주고 종료하지 않는다.

### 8.5 학생 포트폴리오

- 점수표 대신 `나의 생각 기록`을 사용한다.
- 단원마다 세 증거 카드를 비교한다.
- `내가 시도한 것 · 사용한 표현 방법 · AI를 판단한 방법 · 새롭게 할 수 있게 된 것`을 보여준다.
- 배지 대신 학생이 실제로 만든 행동 카드·AAC 메시지·시각 일정표·포스터를 중심으로 보여준다.

### 8.6 선택형 저장 기능

| 기능 | 기본값 | 꺼져 있을 때 |
|---|---|---|
| 차시 진도 이어하기 | 켜짐 | 완료 진도를 다음 접속까지 유지하지 않음 |
| 과정평가 기록 | 꺼짐 | 학생 반응과 과정 증거를 장기 저장하지 않음 |
| 결과물 포트폴리오 | 꺼짐 | 그림·생성 결과를 장기 보관하지 않음 |
| 실제 텍스트 AI | 꺼짐 | 연습용 결과 사용 |
| 이미지 인식·생성 | 꺼짐 | 교과서 예시 결과 사용 |

과정평가를 켜기 전 활동은 소급 복원하지 않는다. 기능을 끌 때는 `새 기록 중단·기존 기록 유지`, `백업 후 삭제`, `즉시 삭제` 중 하나를 교사가 선택한다.

### 8.7 로컬 저장 경계

- 현재 결정대로 한 브라우저에서 학생 한 명을 전제로 한다.
- 동일 기기·동일 브라우저 프로필·동일 교과서 주소에서만 기록을 이어 볼 수 있다.
- 시크릿 모드 종료, 브라우저 데이터 삭제, 기기 초기화 시 기록이 사라질 수 있다.
- 학생 이름 대신 별칭이나 번호 사용을 권장한다.
- 여러 학생이 한 브라우저를 번갈아 사용하기 전 백업과 초기화를 요구한다.

### 8.8 저장 매체

- 설정·진도·구조화된 증거 카드: `localStorage`
- 사용자가 명시적으로 포트폴리오에 담은 학생 그림 또는 AI 생성 축소 이미지: `IndexedDB`
- 원본 입력 이미지·원본 음성: 저장하지 않으며 처리 후 참조를 해제한다.
- AI 연결 키: 기존 방식과 호환하되 백업 파일에는 포함하지 않는다.

### 8.9 백업과 복원

- 교사가 내보내기 시 백업 암호를 설정한다.
- 브라우저의 Web Crypto로 백업 데이터를 암호화한 뒤 파일로 저장한다.
- 복원 시 같은 암호가 필요하며 암호 분실 시 복구할 수 없음을 안내한다.
- 인쇄·PDF는 별칭을 기본값으로 사용하며 민감한 교사 메모 포함 여부를 선택하게 한다.
- 저장 실패 시 학생 활동은 계속하고 `저장되지 않음` 상태와 인쇄·백업 대안을 표시한다.

### 8.10 기존 기록 마이그레이션

현재 6개 일반화 순환 기록은 삭제하지 않는다. 새 18개 증거 카드에 억지로 변환하지 않고 `이전 일반화 기록`으로 보존해 교사 화면에서 별도로 확인한다.

## 9. 교사 운영 안내 허브

### 9.1 접근 흐름

```text
교사용 페이지
→ 비밀번호 확인
→ 교사 시작 안내
→ 기능 선택과 운영 확인
→ 학생 기록·AI·성취기준·데이터 관리
```

현재 교사 비밀번호는 학생의 우발적 접근을 막는 프런트엔드 게이트다. 서버 기반 인증이나 민감정보 보호 경계로 표현하지 않는다.

### 9.2 첫 기록 활성화 확인

과정기록 기능을 처음 켤 때 다음을 확인한다.

- 이 브라우저는 학생 한 명이 사용한다.
- 학생 이름 대신 별칭이나 번호를 사용할 수 있다.
- 기록이 이 브라우저에만 저장됨을 이해한다.
- 브라우저 데이터를 삭제하면 기록이 사라질 수 있음을 이해한다.
- 원본 사진과 음성은 저장하지 않는다.

### 9.3 교사 허브 영역

1. **운영 안내:** 저장 로직, 개인정보, 한계, 수업 전 준비, FAQ
2. **학생 기록:** 68차시 진도, 18개 증거 카드, 관찰 필요 활동, 도움 수준
3. **포트폴리오:** 단원별 결과물, 교사 메모, 인쇄·PDF
4. **AI 연결:** 텍스트·이미지 인식·생성 개별 활성화, 연결 시험, 외부 전송 안내
5. **학습목표·성취기준:** 68차시 목표, 18개 스튜디오 평가 요소, 차시 연결 관계
6. **데이터 관리:** 저장 상태, 마지막 저장, 암호화 백업, 복원, 출력, 전체 삭제, 이전 기록

### 9.4 수업 전 1분 점검

- 올바른 학생 기록이 열려 있는가
- 과정기록을 사용할 것인가
- 실제 AI를 사용할 것인가
- 사용할 이미지에 얼굴·이름·주소가 없는가
- 수업 후 백업이 필요한가

교사 운영 안내는 최초 온보딩 뒤에도 언제든 다시 열 수 있다.

## 10. 청소년형 교과서 에디토리얼 UI

### 10.1 승인된 디자인 방향

‘힙한 앱’보다 실제 중등 교과서의 편집 문법을 디지털 화면에 옮긴다.

- 큰 단원 번호와 비대칭 제목
- 충분한 흰 여백
- 본문·삽화·보조 정보가 교차하는 펼침면
- 둥근 소제목 띠와 여백 메모
- 유기적 이미지 마스크와 작은 말풍선
- 페이지 하단 학습 기록
- 앱 카드의 반복보다 한 장면 안의 정보 위계
- 상호작용을 본문 속 문제와 활동 요소로 통합

참고 교과서를 그대로 복제하지 않고 이러한 편집 원리만 사용한다.

### 10.2 디지털 적응

- PC: 좌우 두 페이지가 연결되는 펼침면을 기본으로 한다.
- 태블릿: 한 페이지 안의 2열 편집으로 전환한다.
- 모바일: 소제목 단위의 단일 열로 재배열한다.
- 인쇄/PDF: 페이지 단위로 자연스럽게 분리한다.
- 진행 도구는 얇은 가장자리 내비게이션으로 두고 본문보다 강하게 보이지 않게 한다.
- 활동 버튼은 큰 앱 카드보다 교과서의 문제 번호·선택 표기·쓰기 공간처럼 표현한다.

### 10.3 언어와 캐릭터

- `보상 받기`, `배지 획득`, `AI 친구와 놀기` 같은 아동용 보상 언어를 축소한다.
- `생각 기록`, `결과물에 담기`, `스튜디오 완료`, `다음 장면`을 사용한다.
- 캐릭터는 화면을 지배하는 마스코트가 아니라 여백에서 질문하거나 주의점을 말하는 동료·안내자 역할로 제한한다.
- 이모지는 핵심 아이콘을 대신하지 않으며 장식적으로 반복하지 않는다.

## 11. 소프트웨어 구조

### 11.1 데이터 모델 경계

기존 `LessonContent`에 모든 기능을 계속 추가하지 않고 다음 책임을 분리한다.

- `LessonRole`: studio, support, module-close
- `StudioDefinition`: 장면, 조건 변화, AI 역할, 결과물, 전이
- `SupportProfile`: 충분한 지원, 보통, 도전적의 정보량·표현·도움 규칙
- `StudioEvidence`: 다섯 핵심 순간과 교사 관찰
- `TeacherRecordingSettings`: 진도·과정평가·포트폴리오·AI 기능 활성화
- `ModuleEditorialTheme`: 단원 색상과 편집 토큰
- `AIConnector`: 텍스트, 이미지 인식, 이미지 생성의 제공자 독립 인터페이스
- `MediaPolicy`: 입력 출처, 개인정보 확인, 저장 금지, 결과 저장 규칙

### 11.2 화면 단위

- `EditorialLessonFrame`: PC 펼침면·태블릿·모바일 레이아웃
- `StudioExperience`: 일곱 단계 스튜디오 오케스트레이션
- `ScaffoldDrawer`: 경험 중 지원 카드
- `MultimodalExpressionInput`: 카드·AAC·글·음성·그림·이미지 선택
- `AiComparison`: 실제/연습 AI 결과, 수용·수정·거절
- `TransferTask`: 새로운 장면 적용
- `EvidenceCard`: 학생용 과정 요약
- `TeacherOnboarding`: 로컬 저장·보안 한계 안내와 활성화 확인
- `TeacherEvidenceDashboard`: 6단원 × 3스튜디오 과정 기록
- `TeacherPortfolio`: 결과물·메모·출력
- `TeacherDataManagement`: 백업·복원·삭제·마이그레이션
- `AIConnectionPanel`: AI 기능별 활성화와 진단

각 화면 단위는 데이터 읽기·쓰기 로직을 직접 소유하지 않고 전용 저장 계층과 훅을 통해 사용한다.

### 11.3 데이터 흐름

```text
학생 입력
→ 개인정보·입력 형식 확인
→ 세션 상태
→ 선택적으로 외부 AI 호출
→ 응답 안전 처리
→ 학생의 비교·수정·전이
→ 과정기록이 켜진 경우 구조화된 증거만 저장
→ 교사 화면과 학생 포트폴리오에서 읽기
```

### 11.4 하위 호환

- 기존 68개 차시 ID와 완료 진도를 유지한다.
- 기존 미션 저장 키를 즉시 삭제하지 않는다.
- 새 스튜디오 엔진이 없는 차시는 기존 렌더러로 폴백한다.
- 단계별 배포 중에도 모든 차시에 접근할 수 있어야 한다.

## 12. 오류와 실패 처리

| 상황 | 학생 화면 | 교사 화면·기록 |
|---|---|---|
| AI 키 없음 | 연습용 예시 결과 사용 | 실제 AI 꺼짐 표시 |
| 네트워크 시간 초과 | 작업 유지, 예시 결과로 전환 | 기술 상세는 교사에게만 표시 |
| 안전 필터 차단 | 다른 표현을 시도하도록 안내 | 차단 사실만 기록, 민감 원문 저장 안 함 |
| 이미지 권한 거부 | 교과서 이미지·글·AAC 대체 | 권한 설정 안내 |
| 개인정보 의심 이미지 | 전송 중단, 교사 확인 요청 | 원본 저장 안 함 |
| 음성 인식 미지원 | 글·AAC로 즉시 전환 | 오류를 학생 실패로 기록하지 않음 |
| localStorage 실패 | 활동 계속, 저장 안 됨 표시 | 백업·출력 대안 제공 |
| IndexedDB 실패 | 결과물 저장 생략 가능 | 구조화된 증거 카드는 유지 시도 |
| 백업 암호 오류 | 복원 중단, 기존 기록 유지 | 반복 실패로 기존 기록을 덮어쓰지 않음 |
| 오래된 기록 형식 | 읽기 전 마이그레이션 또는 이전 기록 표시 | 변환 실패 기록은 원본 유지 |

## 13. 접근성 요구사항

- 모든 조작은 키보드로 가능해야 한다.
- 포커스 순서와 포커스 표시를 명확히 한다.
- 색상만으로 상태·단원·정답을 구별하지 않는다.
- 본문과 주요 버튼은 WCAG AA 대비를 충족한다.
- 이미지에 짧은 설명과 TTS 경로를 제공한다.
- 이미지 활동은 AAC·텍스트·교사 설명으로 대체 수행할 수 있다.
- TTS는 사용자 설정을 따르며 활동 진입만으로 자동 재생하지 않는다.
- STT 실패가 학생 수행 실패가 되지 않는다.
- 모션 감소 설정을 존중한다.
- 한 화면에 기본 행동 하나를 우선 제시하고 나머지는 점진적으로 공개한다.
- 지원 수준을 바꿔도 입력과 진행 상태가 유지된다.
- 모바일에서도 최소 터치 영역과 본문 크기를 유지한다.

- 1~4단원과 6단원의 15개 스튜디오

- 6개 단원 마무리 포트폴리오

- 제공자 독립 AI 연결 계층
- 이미지 안전 게이트

- 실제 이미지 생성 3장면
- 연습용 폴백 결과와 실제 결과 구별
- 네트워크·권한·안전 오류 검증

## 15. 검증 전략

### 15.1 데이터 검증

- 68개 차시에 정확히 하나의 역할이 지정되어야 한다.
- 스튜디오는 정확히 18개, 단원 마무리는 정확히 6개여야 한다.
- 단원마다 스튜디오가 3개여야 한다.
- 모든 스튜디오에 첫 시도·AI 비교·결과물·전이 정의가 있어야 한다.
- 모든 단원 테마에 메인·보조·연한색과 텍스트 대비 토큰이 있어야 한다.

### 15.2 단위 검증

- 과정기록 켜짐·꺼짐 저장 규칙
- 지원 수준 전환 시 상태 보존
- 기존 일반화 기록 보존
- 백업 암호화·복원·암호 오류
- AI 실제 결과와 연습 결과 라벨
- 개인정보 입력 차단과 원본 미저장

### 15.3 브라우저 흐름 검증

- 5단원 첫 스튜디오를 세 지원 수준으로 완주
- AI 연결 성공·키 없음·시간 초과·차단 흐름
- 이미지 권한 허용·거부·개인정보 의심 흐름
- 과정기록 꺼짐 상태에서 활동 완주
- 과정기록 켜짐 상태에서 재접속 후 증거 카드 복원
- 교사 화면에서 3개 스튜디오 비교와 PDF 출력
- 기록 백업, 전체 삭제, 복원

### 15.4 접근성 검증

- 키보드 전용 완주
- 화면 낭독기 제목·랜드마크·상태 안내
- 확대·큰 글씨·모바일 단일 열
- 색각 차이 시 번호·아이콘·텍스트 구별
- TTS 꺼짐 상태의 자동 재생 회귀
- 모션 감소 설정

## 16. 완료 기준

다음 조건을 모두 충족해야 전체 개편을 완료로 본다.

1. 68차시는 유지되고 모든 기존 차시에 접근할 수 있다.
2. 18개 스튜디오가 일곱 단계 학습 문법을 사용한다.
3. 단원마다 3개의 증거 카드와 마무리 포트폴리오가 연결된다.
4. 세 지원 수준이 같은 핵심 경험을 공유하고 활동 중 전환해도 상태가 유지된다.
5. 실제 AI가 없어도 모든 스튜디오를 완료할 수 있다.
6. 이미지 인식 5장면과 이미지 생성 3장면이 안전선과 대체 경로를 갖는다.
7. 원본 사진·음성·영상이 장기 저장되지 않는다.
8. 교사는 기록 기능을 사용하기 전에 저장 로직·한계·보안 경계를 확인한다.
9. 과정기록을 사용하지 않아도 전체 교과서를 학습할 수 있다.
10. 학생 화면이 승인된 교과서 에디토리얼 문법과 6단원 색상 체계를 사용한다.
11. 배지·아동용 보상 언어가 학습 증거·결과물 중심 언어로 정리된다.
12. 기존 진도와 일반화 기록이 삭제되지 않는다.
13. TTS 꺼짐 상태에서는 활동 진입으로 음성이 자동 재생되지 않는다.
14. 주요 흐름의 자동 검증과 실제 브라우저 검증이 통과한다.

## 17. 최종 설계 결정 요약

- 68차시는 유지한다.
- 18개 핵심 경험, 44개 지원 차시, 6개 단원 마무리로 역할을 나눈다.
- 3수준 설명은 공통 경험 안의 지원 체계로 전환한다.
- 모든 스튜디오는 멀티모달 표현을 지원한다.
- 실제 이미지 AI는 제한된 핵심 장면에만 사용하고 항상 연습용 폴백을 둔다.
- 과정평가는 다섯 핵심 순간과 네 교사 관찰 기준을 사용한다.
- 과정기록은 선택 기능이며 한 브라우저·학생 한 명을 전제로 한다.
- 교사 접근 뒤에 상세 운영 안내와 데이터 관리가 존재한다.
- 디자인은 실제 중등 교과서의 편집 문법을 디지털로 재해석한다.
- 여섯 단원은 서로 다른 메인색 카드를 사용하되 공통 종이색과 본문 체계를 공유한다.
- 구현은 5단원 파일럿부터 네 단계로 진행한다.
