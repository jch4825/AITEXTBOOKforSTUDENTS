# 작업자 지침서 ① — 학습목표 수정 (goal-rewrite-guide)

> 이 문서 하나만 보고 작업할 수 있도록 작성됨. 대화 맥락 불필요.
> 대상 작업자: Sonnet급 서브에이전트. 단원(모듈) 하나씩 맡아 반복 수행.

---

## 1. 무엇을 하는 작업인가

발달장애 학생용 AI 교과서의 **학습목표 문장**을 전부 쉬운 행동문으로 바꾼다.

- 수정 파일: `src/data/lessons/hard/m1.ts` ~ `m6.ts` (담당 단원 파일 1개)
- 수정 필드: 각 차시의 `goal: { easy, normal, hard }` — **이 3개 문자열만 수정**한다. concept·terms·method·limits·wrapUpHard는 절대 건드리지 않는다.
- 이 목표는 학생 화면의 "오늘의 목표" 카드에 노출되고, TTS(소리)로도 읽힌다. 난이도 설정에 따라 hard/normal/easy 중 하나가 표시된다 (hard=기본, normal=기초, easy=쉬움).

## 2. 문장 규칙 (반드시 지킬 것)

### 공통
1. **행동문**: "오늘은 ~해 봐요 / ~을 찾아봐요 / ~을 만들어 봐요" 꼴. 학생에게 말 거는 존댓말.
2. **"~할 수 있다" 금지**. 성취기준체·문어체 금지. (교사용 성취기준은 다른 필드에 이미 있음 — 여기선 쓰지 않는다)
3. **그 차시에서 실제로 하는 것만 약속한다.** 목표를 쓰기 전에 반드시 그 차시의 활동을 확인:
   - `src/data/lessons/m{N}.ts`의 같은 차시 `steps` (OX·카드 고르기·짝맞추기·순서·sim-ai·real-ai)
   - `src/data/lessons/hard/m{N}.ts`의 같은 차시 `method` (기본 난이도의 "어떻게 할까요" 절차)
   - 활동에 없는 행동(예: "설명해 봐요"인데 설명할 기회가 없음)은 약속 금지. 애매하면 "알아봐요/찾아봐요/골라봐요"처럼 실제 활동 동사로.
4. **어려운 낱말 금지.** 예외 1개: 그 차시 `terms`에서 가르치는 핵심 용어는 1개까지 그대로 사용 가능 (예: m1-l3에서 '생성형 AI'는 가르치는 말이므로 OK). 그 외의 한자어·추상어(원리, 종합, 예측, 분류, 구별, 파악, 활용 등)는 풀어쓴다: 원리→어떻게 하는지, 분류→나누기, 구별→다른 점 찾기, 활용→써 보기.
5. 한 문장은 짧게. 쉼표 2개 이상 들어가면 두 문장으로 나누거나 줄인다.

### 난이도별 길이
| 필드 | 분량 | 예 |
|---|---|---|
| `hard` (기본) | 행동 1~2개, 1~2문장 | "오늘은 AI가 어디에 있는지 찾아보고, AI가 뭔지 내 말로 말해 봐요." |
| `normal` (기초) | 행동 1개, 1문장 | "우리 주변에서 AI를 찾아봐요." |
| `easy` (쉬움) | 가장 짧은 1문장 | "AI를 찾아봐요." |

세 문장이 서로 다른 길이여야 하고, easy ⊂ normal ⊂ hard 방향으로 내용이 포개져야 한다(쉬움이 기본에 없는 내용을 말하면 안 됨).

## 3. 좋은 예 / 나쁜 예 (실제 차시)

### 예1 — m1-l3 (AI는 어떻게 답해줄까?)
- ❌ (전) hard: "생성형 AI가 학습한 자료를 바탕으로 다음 말을 예측해 답을 만드는 원리를 설명할 수 있다."
  - 문제: 성취기준체, 예측·원리 어려움, "설명"할 활동이 차시에 없음.
- ✅ (후) hard: "오늘은 생성형 AI가 다음 말을 이어 붙여 답을 만드는 모습을 알아보고, 틀린 답도 있는지 살펴봐요."
- ✅ (후) normal: "AI가 어떻게 답을 만드는지 알아봐요."
- ✅ (후) easy: "AI가 답을 만드는 걸 구경해요."

### 예2 — m4-l11 (우리의 안전 약속)
- ❌ (전) hard: "개인정보, 환각, 신고의 뜻을 설명하고 단원 내용을 종합해 안전 약속을 만들 수 있다."
  - 문제: "종합" 어려움, 용어 3개 나열 부담, 성취기준체.
- ✅ (후) hard: "오늘은 배운 안전 약속을 다시 보고, 우리만의 안전 약속을 만들어 봐요."
- ✅ (후) normal: "우리만의 안전 약속을 만들어 봐요."
- ✅ (후) easy: "안전 약속을 만들어요."

### 예3 — 하지 말아야 할 변형
- ❌ "AI의 원리를 이해해 봐요" — '원리·이해'는 관찰 불가능한 추상어. → "AI가 어떻게 하는지 알아봐요"
- ❌ "오늘은 ~을 알아보고, ~을 찾아보고, ~을 만들어 봐요" — 행동 3개는 과부하. 최대 2개.
- ❌ easy가 "생성형 AI를 알아봐요" 인데 hard에 생성형 AI가 없음 — 포함 관계 위반.

## 4. 작업 절차 (단원 1개 기준)

1. `src/data/lessons/hard/m{N}.ts` 전체 읽기 + `src/data/lessons/m{N}.ts`에서 각 차시 `steps`의 활동 종류 확인.
2. 차시 순서대로 goal 3종 재작성. 위 규칙 2·3 준수.
3. 전 차시 수정 후 자가 검사 (아래 체크리스트).
4. `npm run lint && npm run check:encoding` 통과 확인.
5. 커밋: `content(goal): 단원 {N} 학습목표 행동문 전환` + 트레일러 `Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>`.
6. 보고: 차시별 before→after 표 (hard만, easy/normal은 요약) + 판단이 애매했던 차시 명시.

## 5. 자가 검사 체크리스트 (전 차시에 대해)

- [ ] "~할 수 있다"가 한 곳도 없다 (`grep '할 수 있다' src/data/lessons/hard/m{N}.ts` 결과 goal에 0건)
- [ ] 모든 hard가 "~해 봐요/~봐요" 꼴 행동문이다
- [ ] 각 목표의 행동이 그 차시 steps 또는 method에 실제로 존재한다
- [ ] 차시당 어려운 용어는 그 차시 terms의 핵심 용어 1개 이하다
- [ ] easy < normal < hard 길이·포함 관계가 성립한다
- [ ] goal 외의 필드를 건드리지 않았다 (`git diff`로 확인)
- [ ] 소리 내어 읽었을 때 자연스럽다 (TTS로 읽히는 문장임)

## 6. 하지 말 것

- concept/terms/method/limits/wrapUpHard 수정 금지 (목표만!)
- 새 용어 도입 금지, 사전(studentDictionary.ts) 수정 금지
- 차시 순서·키 변경 금지
- "재미있게", "신나게" 같은 수식어 남발 금지 — 행동만 담백하게
